# 客户增长流水线与客户 360 后台信息架构设计

## 1. 背景

当前前端已经从一期“客户增长闭环 demo”扩展到二期“教育服务运营工作台”，包含 CRM、项目/课程、活动、企业助手、学生助手、知识库、报告和系统管理等模块。

现有页面的主要问题不是局部视觉样式，而是信息架构错位：二期能力被平铺成多个同级入口，首页、顶部状态、左侧导航、主内容和右侧上下文同时承载过多信息，导致系统像功能陈列柜，不像围绕业务推进的运营产品。

本设计确认登录后后台采用“客户增长流水线 + 客户 360 工作台”方向。它是 `docs/superpowers/specs/2026-06-10-enterprise-portal-role-workbench-ia.md` 中“后台生产力工具”层的子设计，不是未登录用户看到的系统第一屏。

目标是在保留二期能力的前提下，把登录后的后台体验重新组织为围绕客户增长闭环推进的业务产品原型。

## 2. 设计目标

1. 未登录首屏由公开官网门户承载；本设计只约束登录后的后台工作台。
2. 后台首页不再堆叠全部功能，改为增长总览、今日重点、最近客户和待办。
3. CRM / 客户列表成为客户增长主入口。
4. 点击客户进入客户 360 工作台，由客户详情承接画像、推荐、咨询、跟进、活动和报告。
5. 企业助手、学生助手作为二期扩展入口，不与客户增长主链路抢首屏。
6. OpenAPI、初始化演示数据、fallback 状态、接口状态等 demo 控件集中到系统与演示区域。
7. 页面通过流程、详情页、tabs、抽屉和分区分担复杂度，避免继续把功能平铺进一个工作台。

## 3. 成功标准

1. 登录后用户进入后台首页后能立即理解“今天客户增长该推进什么”，而不是看到全模块陈列。
2. 顾问从客户列表进入客户 360 后，可以完成客户增长主链路的核心演示。
3. 企业助手、学生助手、系统管理和演示控制台仍可进入，但不会干扰客户增长首屏。
4. 不同角色看到不同入口；无权限入口隐藏，页面内不可操作动作以禁用或只读状态展示。
5. 后续新增二期能力时，必须先判断它属于主链路、运营资源、二期助手、报告中心还是系统与演示，不能直接加到首页或一级导航。

## 4. 一级信息架构

登录后后台一级导航如下：

| 一级入口 | 定位 | 包含内容 |
| --- | --- | --- |
| 增长总览 | 首页，回答今天该推进什么 | 增长指标、今日重点、最近客户、待办、最近报告摘要 |
| 客户增长 | 主入口，承载 CRM 流水线 | 客户列表、阶段漏斗、负责人、下一步动作、客户 360 入口 |
| 运营资源 | 支撑客户增长的资源库 | 项目/课程、活动运营、知识库来源和日志 |
| 报告中心 | 管理层决策入口 | 客户经营、员工日报、学生心理健康、投诉处理报告 |
| 二期助手 | 扩展业务入口 | 企业助手、学生助手 |
| 系统与演示 | 治理和演示控制 | 用户角色、权限点、审计、通知、OpenAPI、seed、fallback 状态 |

客户 360 不作为普通一级导航，它由客户列表、最近客户、待办、报告相关客户等入口进入。

## 5. 客户增长主链路

客户增长主链路包括：

1. 客户进入系统。
2. 画像研判。
3. 项目推荐。
4. Dify 咨询记录或 fallback 问答。
5. CRM 跟进和任务。
6. 阶段流转。
7. 活动报名。
8. 报告快照和下一步建议。

该链路是前端重构的优先级最高部分。其他二期能力必须服务或补充该链路，不能反向挤占它的首屏空间。

## 6. 页面结构

### 6.1 增长总览

页面目标：让管理者、顾问和演示者快速看到客户增长状态和今日重点。

主要区域：

| 区域 | 内容 |
| --- | --- |
| 指标栏 | 今日新增、高潜客户、待跟进、活动转化、最近报告 |
| 今日重点 | 需要今天处理的高潜客户、超时跟进、活动邀约 |
| 最近客户 | 最近被创建、咨询、画像或跟进的客户 |
| 待办 | CRM 跟进、活动确认、报告生成等客户增长相关任务 |
| 轻量状态 | 仅保留必要系统状态摘要，详细状态进入系统与演示 |

不在首页常驻展示：企业助手全功能、学生助手全功能、OpenAPI、初始化演示数据、完整 fallback JSON、系统管理表格。

### 6.2 客户增长

页面目标：把 CRM 从普通模块升级为主工作区。

主要区域：

| 区域 | 内容 |
| --- | --- |
| 阶段漏斗 | 新线索、已画像、咨询中、活动邀约、成交/流失 |
| 筛选区 | 关键词、阶段、负责人、推荐项目、创建时间 |
| 客户列表 | 客户、阶段、推荐项目、负责人、最近跟进、下一步动作 |
| 批量视图 | 高潜、超时、活动待邀约、长期培育 |
| 详情入口 | 点击客户进入客户 360 |

该页面不承载所有客户详情，只负责队列管理和进入客户 360。

### 6.3 客户 360

页面目标：围绕单个客户组织所有客户增长动作。

顶部展示：

- 客户姓名、联系方式、阶段、负责人、推荐项目。
- 快捷动作：新增跟进、创建任务、报名活动、生成报告快照。

内部 tabs：

| Tab | 内容 |
| --- | --- |
| 客户概览 | 基础信息、当前阶段、最近跟进、关键缺口、下一步动作 |
| 画像研判 | 结构化画像、匹配分、命中规则、缺失字段、风险提示 |
| 推荐项目 | 推荐项目、命中标签、推荐理由、项目详情入口 |
| 咨询记录 | Dify 问答、引用来源、fallback 状态、conversation id |
| 跟进任务 | 跟进记录、待办任务、完成状态、阶段历史、成交/流失动作 |
| 活动报名 | 推荐活动、报名记录、名单状态、签到结果 |
| 报告快照 | 客户相关报告摘要、经营建议、历史快照 |

右侧上下文面板：

- 仅在客户 360 内出现。
- 默认可收起。
- 内容为 AI 建议、下一步动作、风险提示和缺失信息。
- 不再作为全局常驻侧栏占用所有页面空间。

### 6.4 运营资源

页面目标：维护支撑客户增长的资源，而不是成为独立主链路。

包含：

1. 项目/课程：项目资料、标签、费用周期、适合人群、推荐规则说明。
2. 活动运营：活动创建、报名名单、签到。
3. 知识库：知识来源、同步任务、问答日志、fallback 状态。

运营资源页面可以跳转到客户列表或客户 360，但不应把客户增长动作复制一份。

### 6.5 报告中心

页面目标：承接管理层和经营分析视角。

包含：

1. 客户经营分析。
2. 员工日报汇总。
3. 学生心理健康周报。
4. 投诉处理周报。

客户经营报告应能跳回相关客户或客户 360。学生、日报、投诉类报告属于二期扩展和管理视角，不进入客户增长首屏。

### 6.6 二期助手

页面目标：保留企业助手和学生助手的真实后端能力，但作为扩展业务入口。

企业助手：

| 区域 | 内容 |
| --- | --- |
| 自然语言输入 | 客户录入、客户查询、状态更新、日报、组织架构、新人指南 |
| 快捷指令 | 常用业务指令模板 |
| 结果区 | 创建的客户、查询结果、日报结构化结果、组织信息 |
| 分区 tabs | 客户录入/查询、日报、组织架构/新人指南、受控查询 |

企业助手创建或更新客户后，应提供进入客户增长或客户 360 的路径。

学生助手：

| 区域 | 内容 |
| --- | --- |
| 当前学生 | 学生、项目、老师、状态、风险等级 |
| 服务事项 | 请假、反馈、申请进度、学业节点、生活支持 |
| 老师处理区 | 待审批请假、待处理反馈、心理辅助预警 |
| 分区 tabs | 请假/审批、反馈工单、心理辅助预警、学业与申请进度、生活支持 |

心理相关文案必须坚持“辅助识别”，不得表达专业诊断结论。

### 6.7 系统与演示

页面目标：集中治理能力和 demo 控件，避免污染主业务界面。

包含：

1. 用户、角色、权限点、角色权限绑定。
2. 审计日志和通知中心。
3. OpenAPI 入口。
4. `POST /api/demo/seed` 初始化演示数据。
5. Dify fallback 状态、接口健康和 `/api/phase2/overview`。

系统与演示区域可证明企业级扩展边界，但不作为客户增长首屏内容。

## 7. 角色入口规则

当前阶段采用“前端角色视图 + 后端权限数据展示”，不假装完整登录鉴权已经完成。

| 角色 | 默认入口 | 可见入口 |
| --- | --- | --- |
| 管理员 | 系统与演示或增长总览 | 全部入口 |
| 管理者 | 增长总览 | 增长总览、客户增长、报告中心、二期助手部分管理视图、系统与演示只读治理视图 |
| 顾问 | 客户增长 | 增长总览、客户增长、客户 360、运营资源、报告中心客户经营报告 |
| 员工 | 二期助手 | 企业助手、知识库、有限客户结果入口 |
| 老师 | 二期助手 | 学生助手、学生相关报告、知识库 |
| 学生 | 二期助手 | 学生助手、生活支持问答 |

展示原则：

1. 无权限一级入口直接隐藏，不展示大面积禁用菜单。
2. 当前页面内部分动作无权限时，用禁用态或只读态说明。
3. 系统与演示中展示角色-权限点矩阵，说明当前权限模型。
4. 前端权限只用于体验展示；生产级后端权限校验属于 V2 增强。

## 8. V2 兼容边界

当前前后端代码适合作为 V2 的业务原型和工程底座，但还不是完整生产级企业应用。

已具备的基础：

1. 后端已有 `api / services / models / schemas` 分层。
2. API 统一返回 `{ code, msg, data }`。
3. 企业助手、学生助手、CRM、项目、活动、知识库、报告、权限、审计、通知已有路由和 service 底座。
4. 写操作已具备 service 层和审计思路。
5. Dify、fallback、模板兜底边界适合渐进增强。

后续 V2 需要补强：

1. 前端按业务域拆分模块边界。
2. 权限由前端角色视图升级为后端权限点驱动。
3. 增加登录、Token 和后端接口级权限校验。
4. 引入数据库迁移策略，例如 Alembic。
5. 强化客户、学生、项目、活动、报告、审计和通知之间的数据关系。
6. 补齐部署、环境变量、日志、CI 和运维文档。

## 9. 实施顺序

1. 更新项目规则和原型结构文档，固定前端信息架构约束。
2. 编写 implementation plan，将重构拆成可验证纵向切片。
3. 重构前端壳层：导航、首页、角色入口、系统与演示入口。
4. 建立客户增长列表和客户 360 页面结构。
5. 将画像、推荐、Dify、跟进、活动、报告迁移到客户 360。
6. 整理运营资源、二期助手和系统与演示入口。
7. 运行前端构建和后端测试，按阶段创建中文 Git commit。

## 10. 非目标

本次前端 IA 重构不直接完成以下事项：

1. 不实现完整登录认证。
2. 不引入新的后端权限拦截体系。
3. 不把 SQLite 迁移到 MySQL。
4. 不新增超出 PRD 的业务功能。
5. 不做单纯 CSS 美化来掩盖信息架构问题。
