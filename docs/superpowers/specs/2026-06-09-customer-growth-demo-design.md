# 教育服务客户增长闭环 Demo 设计文档

## 1. 背景与目标

本项目资料来自 `教育服务/` 目录，包含客户需求表、公司信息、公司业务、常见问答、用户画像研判规则、留学政策、测试简历和 SQL 设计草稿。

项目组当前约束如下：

- 开发周期：3 天内完成可演示 demo。
- 团队情况：5 名成员均处于学习阶段，会使用 AI coding，但不适合按专业研发岗位深度拆分。
- 评审对象：技术评审/答辩老师，更关注架构、AI 能力、数据库设计、接口和可扩展性。
- 长期目标：demo 后可继续演进为更完整的企业级交付系统。

本阶段目标不是一次性实现需求表中的全部能力，而是先打穿一条可验证、可解释、可扩展的主链路：

> 客户资料输入 -> 信息抽取 -> 客户画像研判 -> Dify 知识库咨询 -> 项目推荐 -> CRM 线索沉淀 -> 活动报名 -> 智能报告。

## 2. 范围裁剪

### 2.1 三天 demo 范围

本阶段只实现“客户增长闭环”相关能力：

- 客户资料输入与基础解析。
- 客户画像研判。
- Dify 知识库问答。
- 项目推荐。
- CRM 意向客户管理。
- 活动/讲座报名薄闭环。
- 客户经营分析报告。
- OpenAPI 接口展示。
- 演示数据初始化。

### 2.2 薄实现范围

以下能力只做能支撑演示的最小版本：

- 活动报名：预置 2-3 个活动，支持查询和报名。
- 项目库管理：预置新加坡和德国项目，只做查看和推荐。
- LLM Provider：保留真实大模型接入能力，通过环境变量配置；不可用时降级。
- 权限：demo 阶段只保留简单用户/角色概念，不做完整 RBAC。

### 2.3 二期扩展范围

以下需求来自客户需求表，但不进入三天页面开发：

- 企业智能助手：日报、NL2SQL、组织架构、新人指南、学生成绩、投诉管理。
- 学生智能助手：请假审批、心理关怀、售后反馈、学业考务、进度追踪、生活支持、二次转化。
- 智能报告扩展：员工日报汇总、学生心理健康周报、投诉处理周报。
- 完整权限体系、审计日志、消息通知、审批流、Dify 知识库自动同步。

答辩表达方式：

> 三天 demo 先打穿客户增长主链路，企业助手和学生助手作为后续模块扩展。当前设计已保留模块边界和数据扩展方向。

## 3. 总体架构

### 3.1 技术栈

- 后端：FastAPI + SQLAlchemy。
- 数据库：demo 阶段可使用 SQLite，表设计兼容后续 MySQL。
- 前端：React + Vite + TypeScript。
- 知识库检索：统一使用 Dify，Dify 服务和 API Key 由用户提供，项目不负责部署 Dify。
- 大模型：通过统一 LLM Provider 接入真实模型；demo 阶段允许规则引擎和模板生成兜底。
- 接口文档：FastAPI OpenAPI/Swagger。

### 3.2 分层设计

系统分为五层：

1. 前端工作台
   - 客户研判页、Dify 咨询页、CRM 页、活动页、报告页、演示总览页。

2. API 层
   - FastAPI 路由统一返回 `{ code, msg, data }`。
   - 保持接口边界清晰，便于答辩展示。

3. 业务服务层
   - Lead 服务、Profile 服务、Knowledge 服务、Project/Event 服务、Report 服务。

4. AI 编排层
   - Dify 客户端、LLM Provider、规则引擎、模板报告生成器。

5. 数据层
   - 线索、画像研判、项目、活动、报名、问答日志、报告快照。

### 3.3 Dify 边界

Dify 负责：

- 知识库管理。
- 文档切片。
- 检索召回。
- 知识问答。
- 引用来源返回。

本系统负责：

- 业务流程编排。
- Dify API 调用封装。
- 问答结果和引用落库。
- 客户画像研判。
- CRM 和报告生成。

Dify 配置通过环境变量接入：

```env
DIFY_API_BASE=...
DIFY_API_KEY=...
DIFY_APP_ID=...
```

## 4. AI 稳定性策略

### 4.1 规则引擎

客户画像研判以规则引擎为主，确保结果稳定、可解释、可复现。

规则维度包括：

- 年龄。
- 学历。
- 家庭经济条件。
- 留学意向。
- 就业需求。
- 升学需求。
- 移民需求。
- 语言能力。
- 动手能力或专业倾向。

输出内容包括：

- 抽取后的结构化画像。
- 新加坡项目匹配分。
- 德国项目匹配分。
- 推荐项目。
- 命中理由。
- 缺失字段。
- 下一步跟进建议。

### 4.2 模板生成

客户经营分析报告以模板生成为主，确保没有大模型时仍能演示。

报告结构包括：

- 线索总览。
- 项目匹配分布。
- 高潜客户列表。
- 客户共性画像。
- 跟进建议。
- 风险提示。

### 4.3 真实大模型增强

真实大模型作为增强能力，不作为唯一依赖。

可用于：

- 长文本客户资料抽取。
- 报告润色。
- 咨询话术生成。
- 复杂客户背景总结。

降级策略：

- 大模型不可用时，画像研判继续使用规则引擎。
- 大模型不可用时，报告继续使用模板版本。
- Dify 不可用时，Dify 咨询页提示调用失败，但 CRM、画像和报告链路不崩溃。

## 6. API 设计

所有接口统一返回：

```json
{
  "code": 0,
  "msg": "success",
  "data": {}
}
```

错误也使用同一结构：

```json
{
  "code": 40001,
  "msg": "客户不存在",
  "data": null
}
```

### 6.1 客户与画像

- `POST /api/leads`
  - 新建线索。

- `GET /api/leads`
  - 查询线索列表。

- `GET /api/leads/{id}`
  - 查询线索详情。

- `PATCH /api/leads/{id}/status`
  - 更新线索状态。

- `POST /api/profile/assess`
  - 客户画像研判。

### 6.2 Dify 知识问答

- `POST /api/knowledge/chat`
  - 发起知识库咨询。

- `GET /api/knowledge/logs`
  - 查询问答记录。

- `GET /api/knowledge/logs/{id}`
  - 查看单次问答、引用来源和调用状态。

### 6.3 项目与活动

- `GET /api/projects`
  - 查询项目列表。

- `GET /api/projects/recommendations`
  - 获取项目推荐。

- `GET /api/events`
  - 查询活动列表。

- `POST /api/events/{id}/registrations`
  - 给线索报名活动。

### 6.4 报告与演示

- `POST /api/reports/customer-operation`
  - 生成客户经营分析报告。

- `GET /api/reports`
  - 查询报告列表。

- `GET /api/reports/{id}`
  - 查询报告详情。

- `POST /api/demo/seed`
  - 初始化演示数据。

## 8. 演示流程

答辩现场按固定脚本演示：

1. 打开 `/dashboard`，说明系统目标、技术分层、Dify 和规则引擎边界。
2. 打开 `/assessment`，输入或加载样例客户资料。
3. 点击画像研判，展示字段抽取、评分、理由、缺失项和推荐项目。
4. 保存客户为 CRM 线索。
5. 打开 `/chat`，围绕该客户提问公司业务、项目费用、留学政策，展示 Dify 回答和引用。
6. 打开 `/leads`，查看客户列表和状态流转。
7. 打开 `/events`，给客户报名预置活动。
8. 打开 `/reports`，生成客户经营分析报告。
9. 打开 FastAPI OpenAPI，展示接口分组和扩展边界。

- 需求说明。

## 13. 风险与应对

### 13.1 Dify 调用失败

风险：Dify 网络、Key 或应用配置异常。

应对：

- Dify 咨询页显示明确错误。
- CRM、画像、报告链路不依赖 Dify 成功。
- 准备截图或录屏备用。

### 13.2 大模型调用失败

风险：模型额度、网络、Key 或响应超时。

应对：

- 画像使用规则引擎。
- 报告使用模板生成。
- 页面标注“已使用规则/模板降级结果”。

### 13.3 多人 AI coding 冲突

风险：多人同时修改核心结构导致集成失败。

应对：

- 主力控底座。
- 其他成员交低耦合任务包。
- 每天集成，不把集成留到最后一天。

### 13.4 评审认为功能太少

风险：需求表功能项多，demo 只实现客户增长闭环。

应对：

- 明确说明三天 demo 先做可验证主链路。
- 企业助手和学生助手列为二期扩展。
- 用接口边界和数据规划证明扩展路径。

## 14. 设计结论

本设计选择“稳定底座优先”路线。三天内不追求覆盖所有需求项，而是优先交付一条完整、可解释、可演示的客户增长闭环。

该路线符合当前约束：

- 能在三天内完成。
- 适合学习阶段团队。
- 能向技术评审展示架构、AI、数据库和接口。
- 能在后续扩展为企业级系统。
