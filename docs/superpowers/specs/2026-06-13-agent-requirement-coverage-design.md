# Agent 需求覆盖与生产收尾设计

## 1. 目标和边界

本文件用于把 `教育服务/客户需求表.xlsx` 的原始需求，按真人操作逻辑重新核对到当前系统中，并定义 Agent 收尾的最小落地方案。

本轮目标不是重做现有业务线，而是在保留现有页面、API、权限和业务闭环的前提下，补齐“用户能通过对话完成或进入任务”的智能助手体验。

### 1.1 假设

1. 当前系统已具备公开官网、客户增长、CRM、企业助手、学生服务、报告中心、系统治理等基础页面和 API。
2. 本轮优先选择“演示可用优先 + Dify 提前接入”：先产出可导入的 Dify YAML 应用配置，并接入现有问答接口，再让每个核心角色有能问、能跳转、能承接业务的 Agent 入口。
3. Agent 可以与现有功能重叠，但不能替代现有业务页。对话负责理解意图、给出建议、生成草稿、引导跳转；业务页负责完整表单、列表、详情、处理记录和审计。
4. 所有写操作必须由用户确认后执行；AI 输出只能作为建议、草稿、摘要或路由结果。

### 1.2 取舍

1. 先补智能化入口和资料承接，不优先做 BI 大屏、整体视觉重排或大规模导航改造。
2. Dify YAML 和接口接入前置；本地规则和模板只作为 Dify 未配置、调用异常或业务写操作确认前的兜底，不作为最终 Agent 主实现。
3. 先做角色内 Agent，不做一个跨全系统、跨所有权限的万能对话框，避免权限和业务边界失控。

### 1.3 成功标准

1. 每条客户需求都能追踪到：角色、前端入口、后端 API 或资料源、数据表或记录、当前缺口、验收方式。
2. 游客、员工、学生、老师、顾问、管理者至少能在自己的业务场景中看到或使用对应 Agent 能力。
3. “新人入职指引”等资料型需求必须接入真实资料来源，不继续只使用静态假数据。
4. 客户研判和智能报告有明确的 Agent 承接方式：对话可发起、解释、生成草稿或跳转，但最终业务结果仍沉淀到现有对象。
5. 普通业务页面不暴露内部实现话术；系统治理页可以展示知识来源、同步状态、审计和运行状态。
6. 产出 Dify YAML 文件，覆盖客服、企业新人指南、学生生活支持、客户研判和报告解释等核心场景，并能通过现有后端配置接入。

## 2. 总体结论

当前系统不是“没有实现”，而是“页面和 API 已拆开实现，Agent 体验没有统一补齐”。从真人使用角度看，主要缺口如下：

1. `客户需求表/客服Agent` 已有官网客服 Agent，但需要校验 7 类公开咨询是否都能被识别、回答、推荐或引导报名。
2. `客户需求表/企业智能助手` 已有企业助手 API 和部分员工工作台能力，但“新人入职指引”需要接入 `教育服务/公司信息/公司新人指南.docx`，员工对话入口也需要覆盖日报、组织、客户、受控查询。
3. `客户需求表/学生智能助手` 已有学生服务 API 和学生/老师页面，但对话入口要成为“学生自助办理”和“老师处理队列”的入口，而不是只做闲聊。
4. `客户需求表/客户研判` 当前更像顾问页面能力，需要增加“研判 Agent”承接粘贴资料、解释命中规则、推荐下一步。
5. `客户需求表/智能报告` 已有报告生成和报告中心基础，但需要增加“报告 Agent”用于发起生成、解释异常、定位对象、形成管理建议。
6. 当前代码层面，知识问答会先尝试调用 Dify；未配置或异常时走本地兜底并记录状态。企业助手、学生助手也有本地规则和模板能力。因此 Dify YAML 产出与接入需要放到生产基础之后的 Dify 批次，而不是最后收尾。

### 4.2 权限和角色前置

1. 游客只能使用公开客服 Agent，不得访问 CRM、日报、学生风险、审计、权限、系统状态。
2. 员工 Agent 能访问客户快捷录入/查询、日报、组织、新人指南、受控查询；不能查看管理报告和学生心理明细。
3. 学生 Agent 只能访问自己的请假、反馈、成绩、进度、生活支持和心理关怀。
4. 老师 Agent 能处理请假、反馈、心理预警、学业节点和成绩；不能访问顾问 CRM 全量客户。
5. 顾问 Agent 能做客户研判、项目推荐、线索跟进；不能处理学生心理预警。
6. 管理者 Agent 能生成和解释报告；不能承担一线审批和学生提交动作。
7. 管理员只治理用户、角色、权限、知识来源、审计和系统状态；不作为日常业务角色合集。

### 4.3 V3 生产基础前置

如果目标是 V3 生产级交付，而不是只补 Agent 演示能力，以下后端基础必须排在 Dify、角色 Agent 面板和更多业务闭环之前。原因是身份、权限、数据范围和迁移方式会影响几乎所有后续接口；如果后做，前端 token、Agent 上下文、业务写接口和测试都会返工。

| 优先级 | 生产前置项 | 当前风险 | 最合理位置 | 验收方式 |
| --- | --- | --- | --- | --- |
| P0 | MySQL + Alembic 基线 | 当前仍依赖 `create_all()` 和 SQLite 兼容补列；后续新增认证、Agent 日志、Dify 状态字段会增加迁移成本 | 放到所有新增生产字段和新表之前，先建立当前模型的 baseline migration | 空 MySQL 库可通过 Alembic 初始化；SQLite 验收仍可保留 |
| P0 | 生产认证 | 当前依赖演示账号和请求头身份；前端 token 注入、退出、过期处理未成为统一基础 | 放到 RBAC、数据范围、Agent 内部场景之前 | `/auth/login` 返回 token/session；密码哈希；退出和过期处理可验证；生产入口无固定演示密码 |
| P0 | 后端 RBAC 硬化 | 部分接口已有权限依赖，学生服务和知识问答内部场景权限较弱 | 放到所有关键写接口和 Agent 动作之前 | 无权限角色调用关键写接口被拒绝；错误仍保持统一 `{ code, msg, data }` |
| P0 | 数据范围校验 | 角色权限不能替代对象范围；学生、顾问、老师都需要对象级边界 | 放到学生/老师/顾问 Agent 前 | 学生只能看自己，顾问只能处理授权线索，老师只能处理关联学生 |
| P0 | 前端 token 注入基础 | 前端如果先接 Agent，再补 token，会重复改所有请求调用 | 放到前端角色 Agent 面板和后台批量改造前 | `frontend/src/api/client.ts` 统一注入 token，过期跳登录，退出清理状态 |
| P1 | 权限测试矩阵 | 只有成功路径测试会掩盖越权风险 | 和认证/RBAC 同批 | 覆盖关键写接口的允许、拒绝、数据范围三类测试 |

因此，V3 生产级推荐顺序是：

1. Alembic baseline + MySQL 空库初始化。
2. 认证和前端 token 基础。
3. 后端 RBAC 和数据范围校验。
4. 后端 Agent 契约和 Dify YAML/调用接入。
5. 业务闭环补洞、Dify 真实同步、报告导出、部署交付和最终验收。

### 4.4 V3 生产风险与前置决策

以下风险不是新增功能，而是 V3 落地前必须先定下来的工程边界。它们如果后置，会导致认证、权限、Agent、Dify、迁移、部署和验收反复返工。

| 风险 | 当前表现 | 必须提前决策 | 落地约束 | 验收方式 |
| --- | --- | --- | --- | --- |
| 身份来源不统一 | 部分接口仍使用 `actor_username` 或默认 `admin` 表示操作者 | 生产路径统一从 token/session 获取当前用户，请求体中的 `actor_username` 只允许测试或兼容场景使用 | service 层写操作接收解析后的当前用户或用户上下文，不信任前端传入的操作者字段 | 伪造请求体操作者不能越权；审计记录操作者来自 token |
| 数据范围模型不清 | 角色权限存在，但对象归属关系需要进一步明确 | 明确顾问-线索、老师-学生、学生账号-学生档案、管理者-部门范围 | 新增或复用授权字段时先写数据范围规则，再写页面筛选 | 学生不能读取他人数据；顾问不能处理未授权线索；老师不能处理非关联学生 |
| 演示 seed 和生产初始化混淆 | demo seed 能撑演示，但不能代表生产初始化 | 区分生产基础数据、演示数据、测试夹具 | 生产初始化只创建必要角色、权限、管理员或配置；演示数据必须显式触发 | 空库初始化后没有业务假数据；演示 seed 可重复执行且不污染生产验收 |
| Agent 写操作绕过校验 | 当前部分聊天接口会直接创建或更新业务对象 | Dify 和本地 Agent 都只生成草稿或动作建议，确认执行必须走后端权限和数据范围校验 | 统一 `draft -> confirm -> execute -> audit/action_log` | 未确认不写库；确认后有业务记录、审计记录和 Agent 动作记录 |
| 前端权限与后端权限不一致 | 前端隐藏入口不能替代后端校验 | 先产出关键接口权限矩阵，后端按矩阵校验 | 创建、更新、审批、归档、导出、同步、生成报告都必须有权限点 | 直接调用接口也不能越权 |
| Dify 资料与隐私边界 | 公司资料、客户资料、学生资料、心理/投诉内容敏感级别不同 | 将知识来源分成公开资料、内部制度、敏感业务数据、禁止同步数据 | Dify 同步只允许公开资料和授权内部制度；客户简历、学生心理、投诉原文默认禁止同步 | 知识来源有敏感级别；禁止同步资料不能创建同步任务 |
| 配置和密钥泄漏 | Dify Key、JWT secret、数据库连接、CORS 等最终都依赖环境变量 | `.env.example` 只写变量名和说明，不写真实密钥；生产密钥只走部署环境 | 配置读取集中，错误提示不回显密钥值 | 仓库扫描无真实密钥；缺配置时有明确健康检查结果 |
| 测试数据污染 | SQLite、MySQL、seed、测试夹具可能混用 | 区分 SQLite 快速验收、MySQL 空库验收、演示数据验收和单测夹具 | 每类验证使用明确数据库和初始化步骤 | B1-B12 结果标注 SQLite/MySQL；测试可重复运行 |
| 文件上传与导出安全 | 简历、报告导出、Dify 文件同步都会涉及文件路径 | 先定文件目录、扩展名白名单、大小限制、下载权限和清理策略 | 文件不直接使用用户输入路径；导出记录可审计 | 非法扩展名、超大文件、越权下载被拒绝 |
| 后台任务执行方式 | Dify 同步、导出和重试不适合长期阻塞 HTTP 请求 | 先采用轻量任务状态表和轮询；后续可替换为队列 | 接口形态保持“提交任务 -> 查询状态 -> 下载或查看结果” | 长任务失败可见、可重试，不阻塞主业务 |
| 健康检查缺失 | 部署后无法快速判断数据库、Dify、迁移、配置是否可用 | 提前定义健康检查输出，不暴露敏感信息 | 至少检查数据库连接、迁移版本、Dify 配置状态、应用版本 | 部署后可用健康检查判断系统是否可交付 |

这些前置决策应沉淀到第一批实施计划中。实现时可以分阶段落地，但规则不能等到后续页面或 Agent 做完后再补。

建议 YAML 文件位置：

```text

```

YAML 应至少覆盖这些场景：

1. `customer_service`：公司信息、业务、政策、FAQ、项目推荐、活动报名引导、轻量闲聊。
2. `enterprise_guide`：新人指南、组织查询解释、日报草稿说明、员工操作引导。
3. `student_life`：请假/反馈草稿引导、生活支持、心理关怀安全边界、学业考务和申请进度解释。
4. `customer_assessment`：客户资料摘要、画像规则解释、项目推荐理由、跟进建议。
5. `report_assistant`：五类报告生成条件解释、指标解读、异常对象定位建议。

### 4.6 后端 Agent 代码前置

从当前代码排查结果看，现有 `/api/knowledge/chat` 已尝试调用 Dify，未配置或异常时返回本地兜底；`enterprise-assistant/chat` 和 `student-assistant/chat` 仍主要由本地规则直接识别意图并处理。为了避免前端 Agent 面板做完后返工，以下后端代码必须前置完成。

| 优先级 | 后端前置项 | 当前状态 | 为什么必须前置 | 最小动作 | 验收方式 |
| --- | --- | --- | --- | --- | --- |
| P0 | 统一 Agent 请求上下文 | `KnowledgeChatRequest` 只有 `question/scene/lead_id/conversation_id`；企业和学生助手各有独立 schema | Dify YAML、前端 Agent、日志和权限都要依赖同一套 `scene/role/actor/student/lead/context` 约定 | 扩展知识问答请求，补充 `role`、`actor_username`、`student_id`、`business_context`、`action_mode` 等兼容字段 | 旧请求仍可用；新请求能把角色和业务上下文传到服务层 |
| P0 | Dify Client 场景输入契约 | `DifyClient.chat()` 当前只传空 `inputs`，`user` 固定为 `demo-user` | Dify YAML 需要根据场景、角色和业务上下文做路由；如果现在不改，后续每个 Agent 都要重接 | 让 Dify 调用传入 `inputs={scene, role, lead_id, student_id, business_context}`，`user` 使用真实操作者或安全匿名值 | 单测验证 payload 结构；未配置 Dify 时仍可兜底 |
| P0 | 场景权限和数据边界 | 企业、报告接口已有权限依赖；知识问答和学生服务接口权限较弱，公开场景和内部场景混在一个入口 | Dify 接入后，公开客服不能访问内部 CRM、日报、学生风险；内部 Agent 也不能越权 | 增加服务层场景访问策略：`customer_service` 可公开，`enterprise_guide/student_life/customer_assessment/report_assistant` 按角色校验 | 无权限角色访问内部场景返回统一错误；公开客服仍可用 |
| P0 | Agent 动作草稿/确认/执行契约 | 企业聊天当前可直接创建客户和更新状态；学生聊天当前可直接创建请假和心理预警 | 对话框如果直接写库，后续很难补确认、审计和权限，且 Dify 输出不能直接变成写操作 | 统一返回 `requires_confirmation`、`draft`、`action_type`、`confirmation_endpoint`；确认后再调用现有业务 API | 线索、日报、请假、反馈、报告生成都能先预览，再确认写入 |
| P0 | Agent 动作日志 | 已有 `agent_action_log` 表，但当前主要写业务审计和意图日志，Agent 动作记录不统一 | 后续验收要证明“Agent 建议了什么、用户确认了什么、系统执行了什么” | 在确认执行时统一写 `agent_action_log`，并保留业务对象审计 | 动作确认后同时能查到业务记录和 Agent 动作记录 |
| P0 | 知识来源到 Dify 的映射状态 | `knowledge_source` 有 `dify_dataset_id`；同步任务当前只记录未真实同步 | Dify YAML 提前后，必须知道每类资料对应哪个 Dify 数据集或手工导入状态 | 保留手工导入方式，但把来源状态改成可区分：待导入、已导入、需更新、同步失败 | 管理员能看到资料是否已进入 Dify；问答能按场景使用对应数据集 |
| P1 | 报告 Agent 后端解释接口 | 报告生成已有 `/api/reports/generate` 和详情接口，但没有“解释报告/定位异常对象”的统一接口 | 前端报告 Agent 需要稳定后端返回结构，不应只靠前端拼 Prompt | 增加报告解释服务或复用知识问答场景 `report_assistant`，输入报告 ID 和问题 | 管理者可对报告追问，返回指标解释和对象跳转建议 |
| P1 | 客户研判 Agent 后端入口 | 客户画像已有业务能力，但没有统一的对话式研判入口 | 顾问 Agent 需要从资料摘要、规则命中、项目推荐到跟进建议形成链路 | 增加研判场景输入或复用知识问答场景 `customer_assessment`，绑定 lead/assessment | 粘贴资料后能返回摘要、命中规则、推荐项目和下一步 |
| P1 | 契约测试 | 现有测试覆盖本地兜底、企业助手、学生服务和报告，但测试预期仍围绕旧的直接写入行为 | 如果不先改测试，后续每次调整 Agent 都会破坏旧预期或掩盖风险 | 增加 Dify payload、权限、草稿确认、日志的后端测试；同步调整直接写入类测试 | 后端测试能证明配置 Dify 和未配置 Dify 两种路径都稳定 |

不建议作为 Agent 面板第一批前置、但仍属于 V3 生产交付的事项：

1. PDF/Word 导出：属于报告交付增强，不影响认证、权限、Agent 和 Dify 主链路。
2. 复杂 Dify 文件自动同步后台：Dify YAML、资料映射和调用验证可以先做；失败重试、批量同步和健康检查放到 Dify 真实同步批次。
3. BI 大屏：不影响 Agent 能否理解和承接业务。
4. 消息通知真实推送：当前通知表和待办可以支撑演示闭环；真实推送、已读、处理、跳转放到业务闭环批次。
5. 部署脚本和备份恢复：必须在最终交付前完成，但不应阻塞认证/RBAC/Agent 代码闭环。

### 4.7 Agent 行为前置

| 类型 | 规则 |
| --- | --- |
| 问答 | 可以直接回答，但应基于可用资料或业务数据 |
| 推荐 | 可以给出建议，但要提供理由和下一步入口 |
| 草稿 | 可以生成线索、日报、请假、反馈、报告条件等草稿 |
| 写操作 | 必须用户确认后调用 API |
| 查询 | 必须受角色权限和只读白名单限制 |
| 高风险心理内容 | 只做辅助识别、关怀建议和老师跟进，不做诊断 |
| 审计 | 关键写操作、Agent 动作和知识问答调用应记录日志 |

## 5. 完成前置条件的执行顺序

### 5.1 P0：生产基础基线

1. 建立 Alembic baseline，当前模型能在空 MySQL 库初始化。
2. 补 `.env.example`、初始化、回滚和空库验证流程。
3. 实现 `/auth/login`、密码哈希、token/session、退出和过期处理。
4. 前端请求统一注入 token，过期跳回登录。
5. 关键写接口接入后端 RBAC，并补学生、顾问、老师的数据范围校验。
6. 定义身份来源、数据范围、seed 边界、Dify 资料敏感级别、配置密钥、测试隔离、文件安全、任务状态和健康检查规则。

验收：SQLite 和 MySQL 两套基础验证路径明确；无权限或越权访问关键接口会被拒绝。

### 5.2 P0：需求核对和资料登记

1. 用本文件作为需求核对清单，逐项标记已承接、需增强、缺资料、需后置。
2. 在知识来源治理中登记公司信息、公司业务、留学政策、FAQ、新人指南、画像规则。
3. 把 `公司新人指南.docx` 标记为企业新人指南的真实资料源，替换仅靠静态假数据的验收口径。

验收：管理员能在知识来源中看到上述资料类型；业务 Agent 回答时能按场景使用对应资料。

### 5.3 P0.5：后端 Agent 契约与 Dify 接入底座

1. 先扩展后端 Agent 请求上下文，统一 `scene`、`role`、`actor_username`、`lead_id`、`student_id`、`business_context`、`action_mode`。
2. 调整 Dify Client，让 Dify 收到真实场景输入，不再只接收空 `inputs`。
3. 增加场景权限策略，区分公开客服和内部角色 Agent。
4. 统一动作草稿/确认/执行契约，避免聊天接口直接写库。
5. 统一 Agent 动作日志和业务审计写入。

验收：后端测试能证明旧请求兼容，新上下文可传递，内部场景有权限边界，写操作必须确认后执行。

### 5.4 P0.6：Dify YAML 产出与接入

1. 新增 `docs/dify/education-service-agent.yml`，把五类 Agent 场景、系统提示词、安全边界和结构化输出写清楚。
2. 将知识来源治理中的资料分类与 Dify YAML 场景对齐：公司信息、公司业务、留学政策、FAQ、新人指南、画像规则。
3. 调整后端 Dify 调用约定，保证 `/api/knowledge/chat` 可以把 `scene`、角色和业务上下文传给 Dify。
4. 保留本地兜底，但兜底只处理 Dify 未配置、调用异常和演示保底，不再作为最终智能问答主路径。

验收：配置 Dify 后，公开客服和新人指南问答优先返回 Dify 结果；断开 Dify 时主业务不崩溃，并记录调用状态。

### 5.5 P1：统一 Agent 入口设计

1. 保留官网客服 Agent 作为游客入口。
2. 在员工工作台提供企业 Agent 面板，覆盖客户、日报、组织、新人指南、受控查询。
3. 在学生服务台提供学生 Agent 面板，覆盖请假、反馈、心理关怀、学业考务、进度、生活支持。
4. 在老师学生服务工作台提供处理型 Agent 面板，覆盖请假审批、反馈处理、心理预警、学业节点。
5. 在客户增长/客户 360 提供研判 Agent 面板，覆盖资料解析、画像解释、项目推荐、跟进建议。
6. 在报告中心提供报告 Agent 面板，覆盖生成报告、解释报告、定位异常对象。

验收：每个角色进入自己的后台后，能在当前业务上下文中使用对应 Agent，不需要理解系统菜单结构才能完成第一步。

### 5.6 P2：业务动作确认机制

1. 线索创建、状态更新、日报提交、请假提交、请假审批、反馈创建、反馈处理、报告生成都采用“Agent 生成草稿/识别动作 -> 用户确认 -> 调用现有 API”的模式。
2. Agent 回答中必须提供明确按钮或链接：查看详情、确认提交、继续补充、进入页面。
3. 所有确认后的关键动作写入业务记录、处理记录或审计日志。

验收：用户通过 Agent 发起动作后，最终能在对应列表、详情、时间线或报告历史中看到结果。

## 7. 前端形态建议

本轮 UI 优先级最低，建议只做最小增强：

1. 官网：保留现有客服 Agent 面板。
2. 后台：采用右侧抽屉或页面内紧凑面板，不做全局悬浮大机器人。
3. 客户 360、学生详情、报告详情：Agent 优先理解当前对象上下文，减少用户重复输入。
4. 总览页可以保留指标和待办，不急于新增 BI 大屏。
5. 对话结果必须带业务按钮，而不是只输出一段文字。

推荐形态：

| 场景 | 位置 | 形式 |
| --- | --- | --- |
| 游客咨询 | 官网 FAQ/首页 | 页面内客服 Agent |
| 员工工作台 | 右侧或上方紧凑面板 | 企业 Agent |
| 学生服务台 | 页面内服务助手 | 学生 Agent |
| 老师工作台 | 待办区旁边 | 处理型 Agent |
| 客户 360 | 客户详情右侧 | 研判 Agent |
| 报告中心 | 报告列表/详情旁边 | 报告 Agent |
