# Agent 需求覆盖与生产收尾设计

## 1. 目标和边界

本文件用于把 `教育服务/客户需求表.xlsx` 的原始需求，按真人操作逻辑重新核对到当前系统中，并定义 Agent 收尾的最小落地方案。

本轮目标不是重做现有业务线，而是在保留现有页面、API、权限和业务闭环的前提下，补齐“用户能通过对话完成或进入任务”的智能助手体验。

### 1.1 假设

1. 当前系统已具备公开官网、客户增长、CRM、企业助手、学生服务、报告中心、系统治理等基础页面和 API。
2. 本轮优先选择“演示可用优先”：先让每个核心角色有能问、能跳转、能承接业务的 Agent 入口，再逐步增强 Dify 同步和复杂业务动作。
3. Agent 可以与现有功能重叠，但不能替代现有业务页。对话负责理解意图、给出建议、生成草稿、引导跳转；业务页负责完整表单、列表、详情、处理记录和审计。
4. 所有写操作必须由用户确认后执行；AI 输出只能作为建议、草稿、摘要或路由结果。

### 1.2 取舍

1. 先补智能化入口和资料承接，不优先做 BI 大屏、整体视觉重排或大规模导航改造。
2. 先用现有本地资料、现有接口和可解释兜底回答跑通演示，再把资料真实同步到 Dify。
3. 先做角色内 Agent，不做一个跨全系统、跨所有权限的万能对话框，避免权限和业务边界失控。

### 1.3 成功标准

1. 每条客户需求都能追踪到：角色、前端入口、后端 API 或资料源、数据表或记录、当前缺口、验收方式。
2. 游客、员工、学生、老师、顾问、管理者至少能在自己的业务场景中看到或使用对应 Agent 能力。
3. “新人入职指引”等资料型需求必须接入真实资料来源，不继续只使用静态假数据。
4. 客户研判和智能报告有明确的 Agent 承接方式：对话可发起、解释、生成草稿或跳转，但最终业务结果仍沉淀到现有对象。
5. 普通业务页面不暴露内部实现话术；系统治理页可以展示知识来源、同步状态、审计和运行状态。

## 2. 总体结论

当前系统不是“没有实现”，而是“页面和 API 已拆开实现，Agent 体验没有统一补齐”。从真人使用角度看，主要缺口如下：

1. `客户需求表/客服Agent` 已有官网客服 Agent，但需要校验 7 类公开咨询是否都能被识别、回答、推荐或引导报名。
2. `客户需求表/企业智能助手` 已有企业助手 API 和部分员工工作台能力，但“新人入职指引”需要接入 `教育服务/公司信息/公司新人指南.docx`，员工对话入口也需要覆盖日报、组织、客户、受控查询。
3. `客户需求表/学生智能助手` 已有学生服务 API 和学生/老师页面，但对话入口要成为“学生自助办理”和“老师处理队列”的入口，而不是只做闲聊。
4. `客户需求表/客户研判` 当前更像顾问页面能力，需要增加“研判 Agent”承接粘贴资料、解释命中规则、推荐下一步。
5. `客户需求表/智能报告` 已有报告生成和报告中心基础，但需要增加“报告 Agent”用于发起生成、解释异常、定位对象、形成管理建议。

字段说明：

| 字段 | 含义 |
| --- | --- |
| 路径 | 强制按 `客户需求表/大类/细项` 输出 |
| 真人操作 | 用户真实会怎么问或怎么做 |

| 客户需求表/智能报告/投诉处理周报 | 管理者问“投诉主要集中在哪些问题，哪些没解决？” | 老师、管理者 | 反馈工单、报告中心 | 需要 | 报告需要能回到未关闭工单 | 报告列出分类、超时、未关闭和责任人 |

### 4.2 权限和角色前置

1. 游客只能使用公开客服 Agent，不得访问 CRM、日报、学生风险、审计、权限、系统状态。
2. 员工 Agent 能访问客户快捷录入/查询、日报、组织、新人指南、受控查询；不能查看管理报告和学生心理明细。
3. 学生 Agent 只能访问自己的请假、反馈、成绩、进度、生活支持和心理关怀。
4. 老师 Agent 能处理请假、反馈、心理预警、学业节点和成绩；不能访问顾问 CRM 全量客户。
5. 顾问 Agent 能做客户研判、项目推荐、线索跟进；不能处理学生心理预警。
6. 管理者 Agent 能生成和解释报告；不能承担一线审批和学生提交动作。
7. 管理员只治理用户、角色、权限、知识来源、审计和系统状态；不作为日常业务角色合集。

### 4.3 Agent 行为前置

| 类型 | 规则 |
| --- | --- |
| 问答 | 可以直接回答，但应基于可用资料或业务数据 |
| 推荐 | 可以给出建议，但要提供理由和下一步入口 |
| 草稿 | 可以生成线索、日报、请假、反馈、报告条件等草稿 |
| 写操作 | 必须用户确认后调用 API |
| 查询 | 必须受角色权限和只读白名单限制 |
| 高风险心理内容 | 只做辅助识别、关怀建议和老师跟进，不做诊断 |
| 审计 | 关键写操作、Agent 动作和知识问答调用应记录日志 |

## 5. 完成前置条件的执行顺序

### 5.1 P0：需求核对和资料登记

1. 用本文件作为需求核对清单，逐项标记已承接、需增强、缺资料、需后置。
2. 在知识来源治理中登记公司信息、公司业务、留学政策、FAQ、新人指南、画像规则。
3. 把 `公司新人指南.docx` 标记为企业新人指南的真实资料源，替换仅靠静态假数据的验收口径。

验收：管理员能在知识来源中看到上述资料类型；业务 Agent 回答时能按场景使用对应资料。

### 5.2 P1：统一 Agent 入口设计

1. 保留官网客服 Agent 作为游客入口。
2. 在员工工作台提供企业 Agent 面板，覆盖客户、日报、组织、新人指南、受控查询。
3. 在学生服务台提供学生 Agent 面板，覆盖请假、反馈、心理关怀、学业考务、进度、生活支持。
4. 在老师学生服务工作台提供处理型 Agent 面板，覆盖请假审批、反馈处理、心理预警、学业节点。
5. 在客户增长/客户 360 提供研判 Agent 面板，覆盖资料解析、画像解释、项目推荐、跟进建议。
6. 在报告中心提供报告 Agent 面板，覆盖生成报告、解释报告、定位异常对象。

验收：每个角色进入自己的后台后，能在当前业务上下文中使用对应 Agent，不需要理解系统菜单结构才能完成第一步。

### 5.3 P2：业务动作确认机制

1. 线索创建、状态更新、日报提交、请假提交、请假审批、反馈创建、反馈处理、报告生成都采用“Agent 生成草稿/识别动作 -> 用户确认 -> 调用现有 API”的模式。
2. Agent 回答中必须提供明确按钮或链接：查看详情、确认提交、继续补充、进入页面。
3. 所有确认后的关键动作写入业务记录、处理记录或审计日志。

验收：用户通过 Agent 发起动作后，最终能在对应列表、详情、时间线或报告历史中看到结果。

## 6. 最小落地方案

### 6.1 第一批：公开客服 Agent 和企业新人指南

优先级最高，因为这两项最容易在演示中被客户直接发现。

范围：

1. 校验官网客服 Agent 覆盖公司信息、业务、政策、项目推荐、活动报名、FAQ、闲聊。
2. 将真实新人指南资料作为企业 Agent 和新人指南入口的资料源。
3. 在员工工作台中补一个轻量企业 Agent 面板，优先支持新人指南、组织查询、日报草稿。

不做：

1. 不重构官网首页。
2. 不做复杂 Dify 文件同步后台。
3. 不改动客户增长和学生服务的现有主链路。

### 6.2 第二批：学生和老师 Agent

范围：

1. 学生 Agent 支持请假草稿、反馈草稿、进度查询、考务查询、生活支持、心理关怀。
2. 老师 Agent 支持待审批请假、待处理反馈、心理预警、学业节点查看。
3. 学生心理风险输出必须带安全边界。

不做：

1. 不让学生 Agent 直接替老师审批。
2. 不把心理关怀包装成医学诊断。

### 6.3 第三批：客户研判 Agent 和报告 Agent

范围：

1. 客户研判 Agent 支持粘贴客户资料、解释画像规则、推荐项目、生成跟进建议。
2. 报告 Agent 支持生成五类报告、解释指标、定位异常对象、给出管理建议。

不做：

1. 不新增复杂 BI 大屏作为前置条件。
2. 不让报告 Agent 直接修改业务数据。

## 7. 前端形态建议

本轮 UI 优先级最低，建议只做最小增强：

1. 官网：保留现有客服 Agent 面板。
2. 后台：采用右侧抽屉或页面内紧凑面板，不做全局悬浮大机器人。
3. 客户 360、学生详情、报告详情：Agent 优先理解当前对象上下文，减少用户重复输入。
4. 总览页可以保留指标和待办，不急于新增 BI 大屏。
5. 对话结果必须带业务按钮，而不是只输出一段文字。

推荐形态：

| 场景 | 位置 | 形式 |
| --- | --- | --- |
| 游客咨询 | 官网 FAQ/首页 | 页面内客服 Agent |
| 员工工作台 | 右侧或上方紧凑面板 | 企业 Agent |
| 学生服务台 | 页面内服务助手 | 学生 Agent |
| 老师工作台 | 待办区旁边 | 处理型 Agent |
| 客户 360 | 客户详情右侧 | 研判 Agent |
| 报告中心 | 报告列表/详情旁边 | 报告 Agent |
