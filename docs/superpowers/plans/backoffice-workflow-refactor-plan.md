# 后台业务动线重构实施计划

## 1. 计划定位

本计划用于执行后台业务动线重构。它不替代 PRD、原型结构和需求覆盖矩阵，而是把已经确认的方向拆成可验证、可提交的实施批次。

核心原则：

1. 保留当前后台视觉风格，不换 UI 皮肤。
2. 角色工作台只做总览入口。
3. 具体业务功能拆成独立业务页。
4. 每个业务页具备列表/队列、筛选、详情、操作、状态和历史。
5. 不丢失客户需求表、PRD 和原型结构中的任何需求。

| `docs/design/后台操作反馈统一标准-v1.md` | 操作反馈标准 |

## 3. 总验收标准

1. 所有需求均能追踪到角色、前端入口、API、数据表和验收标准。
2. 工作台总览和左侧导航都进入同一个独立业务页。
3. 每个独立业务页都使用当前后台视觉风格。
4. 页面布局紧凑、少空白、文字不挤压。
5. 创建、查看、详情、处理、状态回显、历史记录形成闭环。
6. 刷新后保留当前角色、页面和关键对象 ID。
7. 普通业务角色不出现实现话术。
8. 每批运行：
   - `cd backend && python -m pytest -v`
   - `cd frontend && npm.cmd run build`
9. 每批独立中文 commit。

## 4. 批次一：学生服务闭环样板

目标：

- 建立独立业务页样板。
- 解决当前学生服务和老师学生服务页面混杂问题。
- 用请假和反馈验证跨角色闭环。

范围：

| 页面 | 目标 |
| --- | --- |
| 学生请假申请 | 新建申请、我的请假、详情、撤销/补充、处理记录 |
| 老师请假审批 | 待审批队列、筛选、详情、同意/驳回、审批历史 |
| 学生反馈提交 | 新建反馈、我的反馈、详情、补充说明、处理状态 |
| 老师反馈处理 | 工单队列、筛选、详情、回复、关闭、归档、处理记录 |

建议文件：

- `frontend/src/navigation.ts`
- `frontend/src/App.tsx`
- `frontend/src/pages/BackofficeShellPage.tsx`
- `frontend/src/pages/StudentServicePage.tsx`
- `frontend/src/pages/TeacherStudentServicePage.tsx`
- 必要时新增学生服务子页面组件
- 如 API 字段不足，最小修改 `backend/app/services/student_assistant_service.py` 和对应 schema/routes/tests

验收：

1. 学生提交请假后，老师请假审批队列可见。
2. 老师同意/驳回后，学生端刷新可见最新状态和处理记录。
3. 学生提交反馈后，老师反馈处理队列可见。
4. 老师回复、关闭或归档后，学生端刷新可见处理状态和记录。
5. 页面保留当前后台视觉，右侧无大片空白。

Commit 建议：

- `重构学生服务请假反馈动线`

## 5. 批次二：成绩、学业和申请进度

目标：

- 将学生成绩查询、老师成绩录入、申请进度和考务节点拆成独立业务页。

范围：

| 页面 | 目标 |
| --- | --- |
| 学生成绩查询 | 学生只读查看成绩、课程、老师反馈 |
| 老师成绩录入 | 按学生/课程筛选，录入和修改成绩，保留修改记录 |
| 学生申请进度 | 查看文书、院校、签证等阶段和下一步 |
| 老师学业/进度 | 查看学生列表、考务节点、申请进度和风险 |
| 学生考务节点 | 查看考试、DDL、材料提醒 |

验收：

1. 老师录入成绩后，学生端成绩查询同步显示。
2. 学生申请进度刷新后仍停留当前阶段。
3. 老师学业/进度页不混入请假和反馈操作。

Commit 建议：

- `重构成绩学业和申请进度动线`

## 6. 批次三：顾问客户增长和 CRM

目标：

- 将顾问增长功能拆成可操作的独立业务页。

范围：

| 页面 | 目标 |
| --- | --- |
| 新建线索 | 手动录入、口述草稿、资料来源、保存后定位 |
| 线索队列 | 关键词、状态、负责人、来源、时间范围筛选，漏斗节点联动 |
| 客户 360 | 资料、画像、推荐、跟进、任务、活动、时间线 |
| 跟进任务 | 创建、处理、完成、历史记录 |
| 活动邀约 | 活动列表、客户报名状态、邀约或报名 |

验收：

1. 顾问手动录入线索入口显著。
2. 点击漏斗节点后列表和数量同步变化。
3. 客户 360 刷新后仍保持客户 ID。
4. 跟进、任务、阶段流转写入时间线。

Commit 建议：

- `重构顾问客户增长和CRM动线`

## 7. 批次四：员工工作台和企业智能能力

目标：

- 将企业智能助手能力拆入员工日常任务，而不是集中放在助手页。

范围：

| 页面 | 目标 |
| --- | --- |
| 客户快捷录入 | 口述/文本转草稿，确认后入库 |
| 客户查询/状态更新 | 受控查询和轻量状态更新 |
| 日报提交 | 口述日报、结构化草稿、确认提交 |
| 日报/周报 | 列表、详情、日期/员工/部门筛选 |
| 组织查询 | 部门、职责、同事联系方式和详情 |
| 新人指南 | 制度和流程问答 |
| 受控查询 | NL2SQL 只读白名单和阻断写 SQL |

验收：

1. AI 生成结果必须确认后写库。
2. 组织查询和日报不混在同一个杂乱面板。
3. 写 SQL 或危险查询被阻断。

Commit 建议：

- `重构员工工作台和企业智能动线`

## 8. 批次五：管理者经营报告

目标：

- 管理者后台按经营、日报、心理、投诉和风险队列组织。

范围：

| 页面 | 目标 |
| --- | --- |
| 经营总览 | 增长指标、风险摘要、报告入口 |
| 增长总览 | 线索、转化、流失、活动 |
| 日报汇总 | 日报/周报、部门/员工筛选 |
| 心理周报 | 心理风险趋势和跟进状态 |
| 投诉周报 | 投诉数量、分类、处理时效、未关闭事项 |
| 风险队列 | 统一待处理风险 |

验收：

1. 报告生成后自动定位最新报告详情。
2. 管理者不看到 OpenAPI、seed 写操作。
3. 老师只看到学生服务相关报告，管理者看经营汇总。

Commit 建议：

- `重构经营管理报告动线`

## 9. 批次六：管理员治理

目标：

- 将系统状态、权限、审计、通知、知识来源、OpenAPI、seed 和 fallback 统一归入管理员治理。

范围：

| 页面 | 目标 |
| --- | --- |
| 用户管理 | 用户列表、详情、状态 |
| 角色权限 | 角色、权限点、绑定 |
| 审计日志 | 关键操作筛选和详情 |
| 通知管理 | 通知和待办 |
| 知识来源 | 来源、同步任务、fallback 状态 |
| 系统状态 | OpenAPI、seed、接口状态 |

验收：

1. 普通业务角色不出现 OpenAPI、seed、fallback、权限矩阵等实现话术。
2. 管理员治理动作有审计或任务记录。
3. 权限接口继续返回统一 `{ code, msg, data }`。

Commit 建议：

- `重构系统治理后台动线`

## 10. 禁止事项

1. 不把左侧导航改成窄侧栏模板风格。
2. 不替换当前后台视觉基准。
3. 不一次性重写全站 UI。
4. 不在一个角色页面里继续堆多个完整业务动线。
5. 不删除已验证的 API 能力。
6. 不提交 `教育服务/` 原始资料。
7. 不把 OpenAPI、seed、fallback、权限矩阵暴露给普通业务角色。
8. 不让 AI 输出直接写入关键业务表。

## 11. 每批执行流程

每批实施时按以下顺序：

1. 对照 `backoffice-requirement-traceability.md` 确认需求范围。
2. 检查当前 git 状态。
3. 只修改本批相关文件。
4. 先补或调整测试，再改实现。
5. 运行后端测试。
6. 运行前端构建。
7. 用浏览器手动检查关键链路。
8. 只 stage 本批相关文件。
9. 创建中文 commit。
