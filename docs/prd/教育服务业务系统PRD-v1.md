# 教育服务业务系统 PRD v1

## 1. 文档信息

- 产品名称：教育服务业务系统
- 当前版本：PRD v1
- 当前阶段：二期产品定义
- 资料来源：`教育服务/客户需求表.xlsx`、公司信息、公司业务、留学政策、用户画像规则、一期实现结果、二期 implementation plan
- 目标读者：项目负责人、开发 Agent、答辩评审、产品/测试协作者

## 2. 产品定位

本系统定位为面向教育服务机构的“官网获客门户 + 运营与智能助手一体化平台”。系统不只是一个客服问答工具，也不只是内部工作台，而是对外通过企业官网展示公司背景、业务服务、项目活动和咨询入口，对内围绕客户增长、员工协同、学生服务、知识库问答和经营报告形成业务闭环。

一期已经完成客户增长闭环 demo：客户资料输入、画像研判、Dify 咨询、CRM 沉淀、活动报名和客户经营报告。二期在此基础上升级为完整业务系统，前端入口采用“公开官网门户 -> 登录入口 -> 按角色进入后台生产力工具”的结构，并增加企业助手、学生助手、权限角色、完整 CRM、项目课程管理、活动运营、报告中心和知识库增强。

公开官网参考同类教育服务网站的信息组织方式，但不复制具体视觉和文案。官网首页的产品责任是建立信任、解释服务、承接咨询，不展示后台功能清单；后台才展示 CRM、客户 360、助手、权限和演示控制等内部生产力能力。

## 3. 产品目标

### 3.1 业务目标

1. 提升客户转化效率：从咨询、研判、推荐、跟进到活动报名形成可追踪闭环。
2. 降低员工操作成本：员工可通过企业助手完成客户录入、查询、日报、组织信息和内部知识问答。
3. 提升学生服务体验：学生可自助完成请假、反馈、进度查询、学业提醒和生活支持咨询。
4. 提升管理决策效率：管理层通过报告中心查看客户经营、员工日报、心理健康和投诉处理情况。
5. 提升交付可信度：通过权限、审计、知识来源、日志和 fallback 证明系统具备企业级扩展边界。
6. 提升公开获客能力：未登录访客可通过官网了解企业背景、业务服务、项目课程、活动讲座、FAQ 和咨询方式。
7. 提升官网转化效率：游客能在首页快速完成“了解企业 -> 判断服务是否匹配 -> 查看项目/活动 -> 发起咨询或报名”的路径。

### 3.2 产品成功标准

1. 未登录用户默认进入公开官网门户，而不是直接进入后台工作台。
2. 官网至少覆盖企业介绍、业务服务、信任背书、项目/课程展示、活动/讲座、知识/FAQ、联系我们和登录入口。
3. 登录页可按管理员、管理者、顾问、员工、老师、学生等角色进入对应后台入口。
4. 客户增长主链路不被破坏，并在登录后后台通过客户增长和客户 360 演示。
5. 企业助手至少打通“自然语言录入客户、口述日报、日报汇总、组织架构查询、新人指南问答”。
6. 学生助手至少打通“请假申请审批、反馈工单、心理风险识别记录、学业节点查询、申请进度查询”。
7. 报告中心至少生成四类报告：客户经营分析、员工日报汇总、学生心理健康周报、投诉处理周报。
8. 权限和审计具备可演示闭环：不同角色可看到不同入口，关键操作有审计记录。
9. 所有 API 保持 `{ code, msg, data }` 响应结构。

## 5. 业务主线

### 5.1 公开获客主线

未登录访客访问官网后，先了解企业背景、业务服务、项目/课程、活动/讲座和 FAQ。访客可通过咨询表单、活动报名或知识问答留下需求，系统后续可将有效咨询沉淀为 CRM 线索，由顾问进入客户增长主链路跟进。

公开获客主线必须从游客视角表达价值：服务路径、适合人群、项目周期、费用区间、活动场景和联系方式优先；内部画像评分、CRM 阶段、权限、审计、OpenAPI 和 fallback 调试信息不得在官网展示。

### 5.2 客户增长主线

客户资料进入系统后，先进行画像研判和项目推荐，再沉淀为 CRM 线索。顾问通过 CRM 跟进记录、任务和阶段流转推进转化；活动运营提供讲座报名入口；报告中心定期汇总客户经营情况。

### 5.3 企业助手主线

员工通过企业助手用自然语言完成低成本操作，包括新增客户、查询客户、更新状态、提交日报、查询组织架构、新人指南问答和受控数据查询。企业助手不是直接写数据库的大模型，而是“意图识别 + 服务层操作 + 审计记录”。

### 5.4 学生助手主线

学生通过学生助手提交请假、投诉建议、查询学业节点、查询申请进度和咨询海外生活知识。系统对心理倾诉做风险辅助识别，触发预警后交由老师跟进，不替代专业心理诊断。

### 5.5 管理报告主线

系统把 CRM、日报、反馈、心理预警等数据转化为报告快照，管理层通过报告中心查看趋势、风险和建议。报告以模板和规则兜底，真实大模型只做润色或摘要增强。

## 7. 模块需求

### 7.1 完整 CRM

目标：让顾问能围绕客户完成从初次咨询到签约或流失的全过程管理。

核心需求：
- 新建和编辑线索。
- 查看客户详情、画像结果、推荐项目、问答记录、活动报名。
- 新增跟进记录，记录沟通方式、内容和下一步动作。
- 创建待办任务，支持完成状态。
- 更新客户阶段，并保存阶段流转历史。
- 标记成交或流失，成交记录转化路径，流失记录原因。

验收标准：
- `GET /api/leads/{id}` 返回客户基础信息和当前状态。
- `GET /api/leads/{id}/timeline` 返回画像、问答、跟进、活动、状态历史。
- 前端 CRM 页面支持列表、详情和跟进操作。

### 7.2 客服 Agent

目标：面向客户提供公司信息、业务、政策、项目、活动和 FAQ 问答。

核心需求：
- 识别客户咨询意图：公司信息、业务查询、政策查询、项目推荐、活动报名、FAQ、闲聊。
- 基于 Dify 返回知识库答案和引用来源。
- Dify 未配置时返回 fallback 答案和状态。
- 对项目推荐和活动报名调用系统业务 API，而不只返回文本。

验收标准：
- `POST /api/knowledge/chat` 可记录问答日志。
- 问答结果包含 `status`，可区分真实 Dify 和 fallback。
- 客服页面能展示引用来源或 fallback 原因。

### 7.3 企业助手

目标：为员工提供低成本业务操作入口。

核心需求：
- 意向客户录入：员工输入或口述客户信息，系统结构化并创建 CRM 线索。
- 意向客户查询：按姓名、状态、负责人查询客户。
- 客户状态更新：员工自然语言指令可映射到受控状态更新。
- 口述日报：把员工输入转成日报正文、核心进展、风险和下一步计划。
- 管理日报查阅：按日期、员工、部门汇总日报。
- 组织架构查询：查询部门、负责人和联系方式。
- 新人指南：基于 Dify 或 fallback 回答制度和入职流程问题。
- NL2SQL：仅允许查询白名单表，不允许写操作。

验收标准：
- 企业助手对话记录落库。
- 至少 5 类意图有明确处理结果。
- 所有写操作经过 service 层，不由模型直接写库。

### 7.4 学生助手

目标：为学生提供服务自助入口，为老师提供跟进闭环。

核心需求：
- 请假申请：学生提交原因、开始时间、结束时间。
- 请假审批：老师同意或拒绝，系统记录审批人和时间。
- 心理关怀：学生可进行情绪表达，系统辅助识别风险等级。
- 心理预警：高风险内容生成预警，老师跟进后可关闭。
- 售后反馈：学生提交投诉或建议，系统生成工单。
- 学业考务：学生查询考试、论文 DDL、材料提交节点。
- 申请进度：学生查询文书、院校申请、签证等阶段状态。
- 生活支持：通过知识库回答海外医疗、交通、紧急求助等问题。
- 增值转化：根据学生意向推荐升学项目，但必须可关闭或明确标注为推荐。

验收标准：
- 学生提交事项后状态可追踪。
- 老师处理后学生可看到结果。
- 心理预警文案只表达辅助识别，不作诊断。

### 7.5 项目/课程管理

目标：把新加坡、德国、语言培训、研学、背景提升等项目结构化，支撑推荐和运营。

核心需求：
- 项目基础信息：名称、国家、类别、适合人群、费用区间、周期、招生条件。
- 项目标签：升学、就业、移民、低成本、短学制、带薪实习等。
- 项目方案：可配置不同学制或路径，如 2+2、0.5/1+2、德国双元制。
- 推荐规则说明：展示客户为什么被推荐该项目。

验收标准：
- 项目页面可查看列表和详情。
- 画像推荐能展示项目命中标签和理由。

### 7.6 活动运营

目标：用活动承接客户和学生运营。

核心需求：
- 活动创建：名称、类型、时间、地点、人数上限、适合对象。
- 活动报名：支持线索和学生两类主体。
- 报名名单：查看报名人、来源、状态。
- 签到：记录签到时间和操作人。

验收标准：
- 报名后活动人数变化。
- 报名名单和签到记录可查。

### 7.7 报告中心

目标：把业务数据转化为管理层可读的结构化报告。

核心需求：
- 全域客户经营分析报告：新增、成交、流失、项目分布、高潜客户、跟进建议。
- 员工日报汇总报告：日/周维度汇总进展、产出、风险和待协调事项。
- 学生心理健康周报：风险学生、情绪标签、趋势、跟进建议。
- 投诉处理周报：投诉数量、类型、处理时效、未决风险。

验收标准：
- 每类报告都能生成 `report_snapshot`。
- 报告详情可查看结构化 JSON 内容。

### 7.8 权限、审计和通知

目标：让系统具备企业级交付的基础治理能力。

核心需求：
- 角色：管理员、管理者、顾问、老师、学生。
- 权限点：菜单访问、数据查看、操作权限。
- 审计日志：记录关键写操作。
- 通知：记录待办、审批、反馈处理等提醒。

验收标准：
- 前端至少按角色展示不同菜单分组。
- 后端关键写操作能写审计日志。

## 8. 信息架构

系统采用三层入口：

```text
公开官网门户 -> 登录入口 -> 登录后后台生产力工具
```

### 8.1 公开官网门户

未登录用户默认进入公开官网门户。

一级导航：

- 首页
- 企业介绍
- 业务服务
- 项目/课程
- 活动/讲座
- 知识/FAQ
- 联系我们
- 登录

官网首页建议组织为：

1. 首屏：企业定位、服务对象、核心承诺、咨询 CTA、登录后台入口。
2. 服务区：留学规划、国际本科、德国双元制、语言培训、背景提升、学生服务。
3. 信任区：服务流程、项目资源、顾问/老师能力、可追踪服务体系。
4. 转化区：热门项目、近期活动、FAQ 摘要、联系方式。

官网不得展示 CRM 客户列表、客户 360、员工日报、学生心理预警明细、审计日志、权限矩阵、OpenAPI、seed 和接口健康等内部信息。

### 8.2 登录入口

登录页连接官网与后台。当前 V1 先支持演示登录和角色跳转：

- 管理员 -> 系统与演示 / 系统管理
- 管理者 -> 增长总览
- 顾问 -> 客户增长
- 员工 -> 企业助手
- 老师 -> 学生助手老师视图
- 学生 -> 学生助手学生视图

真实账号密码、Token、会话管理和后端接口级权限校验归入 V3 生产化增强。

### 8.3 登录后后台生产力工具

后台一级导航：

- 增长总览
- 客户增长
- 运营资源
- 报告中心
- 二期助手
- 系统与演示

客户 360 不作为普通一级导航，由客户列表、最近客户、待办或报告关联客户进入。

后台分组：

| 分组 | 包含内容 |
| --- | --- |
| 增长总览 | 今日重点、最近客户、待办、经营状态 |
| 客户增长 | CRM 流水线、客户列表、阶段推进、客户 360 入口 |
| 客户 360 | 客户概览、画像研判、推荐项目、咨询记录、跟进任务、活动报名、报告快照 |
| 运营资源 | 项目/课程、活动运营、知识库 |
| 报告中心 | 客户经营、员工日报、学生心理健康、投诉处理报告 |
| 二期助手 | 企业助手、学生助手 |
| 系统与演示 | 用户角色、权限点、审计日志、通知中心、OpenAPI、seed、fallback、接口状态 |

## 9. API 分组规划

## 10. 数据原则

1. 一期表继续保留，二期优先新增表扩展。
2. 业务状态必须可追踪，涉及审批和阶段变化的动作要记录历史。
3. AI 输出只作为建议或结构化输入来源，关键写库必须通过服务层校验。
4. 知识库来源、问答日志、报告快照都要可追溯。
5. 未建立迁移工具前，不做破坏性字段变更。

## 11. AI 能力边界

| 能力 | 允许做 | 不允许做 |
| --- | --- | --- |
| Dify 问答 | 知识检索、引用来源、FAQ、政策说明 | 替代业务状态判断 |
| 画像研判 | 规则评分、推荐理由、缺失字段提示 | 直接决定签约或拒绝客户 |
| 企业助手 | 意图识别、摘要、受控业务调用 | 直接执行任意 SQL 或绕过 service |
| 学生助手 | 情绪辅助识别、生活知识问答 | 做心理诊断或医疗建议 |
| 报告生成 | 模板汇总、趋势说明、风险提示 | 编造不存在的数据 |

## 12. 分期路线

### V1：官网门户 + 角色后台可演示系统

目标：先形成真实企业级产品入口：未登录用户看到官网，登录后按角色进入后台，并保留二期核心模块的可操作闭环。

交付：
- 公开官网门户。
- 登录页和角色跳转。
- 登录后后台壳层。
- CRM 详情和跟进。
- 企业助手基础意图。
- 学生助手基础服务。
- 项目/课程、活动运营、报告中心、权限审计的最小可用版本。

### V2：完整运营后台和公开内容管理

目标：补齐日常使用所需的 CRUD、筛选、状态流转、报告历史，并让公开官网内容具备后台维护能力。

交付：
- 官网项目、活动、FAQ 内容维护。
- 完整项目管理。
- 复杂 CRM 筛选。
- 日报周报。
- 投诉处理和心理预警跟进。
- 知识来源管理。

### V3：生产化增强

目标：把演示系统增强为可部署、可维护、可多人使用的系统。

交付：
- 登录认证。
- 后端权限校验。
- Alembic 迁移。
- MySQL 配置。
- Dify 真实同步。
- 消息通知。
- 部署文档。

## 13. 非功能需求

1. 可演示性：任意核心模块应有 seed 数据或 fallback 数据。
2. 可追溯性：关键 AI、报告、审批和状态流转有日志或快照。
3. 可扩展性：新增模块按 `models/schemas/services/api` 分层。
4. 可测试性：每个阶段有后端测试和前端构建验证。
5. 安全性：不提交密钥、数据库文件、`教育服务/` 原始资料和本地构建产物。

## 14. 原型交付范围

原型先做中低保真结构，不追求视觉高保真。目标是定义页面、区域、核心控件和主流程，为后续 React 可点击原型提供蓝图。

原型说明文档：

- `docs/prd/教育服务业务系统原型结构-v1.md`

## 15. 当前不做

1. V1 不做完整生产认证，只做演示登录和角色跳转。
2. 不做真实短信、企业微信、邮件推送。
3. 不做完全自动化 Dify 数据集切片上传。
4. 不做任意 SQL 生成执行。
5. 不做心理诊断、医疗建议或法律结论。
6. 不做完整财务、合同、支付和电子签章。

## 16. 待确认问题

1. 官网公开项目、活动、FAQ 的内容维护是否在 V2 做后台管理，还是 V1 先使用静态配置和已有接口。
2. 公开咨询表单提交后是否在 V1 直接创建 CRM 线索，还是先做前端提交成功状态。
3. 学生端是否需要在 V2 单独做移动端风格，还是继续放在统一后台中按角色展示。
4. 生产级登录认证、Token、后端权限校验的具体方案在 V3 使用自建认证还是接入第三方身份服务。
5. 报告是否需要导出 PDF/Word，还是先保留页面和 JSON 快照。
