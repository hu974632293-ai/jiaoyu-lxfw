# 业务动线与数据同步测试计划

## 测试目标

在继续后台视觉和 UI 精修前，先确认核心业务对象在创建方、处理方、管理方之间能稳定同步，刷新页面后仍能恢复当前业务位置，普通业务角色页面不暴露实现话术。

## 测试前准备

1. 启动后端和前端，确认浏览器访问 `http://localhost:5173/`。
2. 如需要重置演示数据，只在管理员治理入口执行初始化。
3. 每条业务动线测试时记录：创建入口、对象名称或编号、处理入口、管理侧回显位置、刷新后的 URL。
4. 每发现问题先判断类型：数据不同步、URL 丢失、权限绕过、操作反馈不清楚、布局遮挡。

## 入口与 URL 恢复

| 编号 | 测试项 | 操作 | 通过标准 |
| --- | --- | --- | --- |
| U1 | 公开官网子页恢复 | 依次进入关于、服务、项目、活动、FAQ、联系页后刷新 | 刷新后仍停留在当前公开子页 |
| U2 | 角色默认后台 | 分别以顾问、员工、老师、学生、管理者、管理员进入后台 | 进入各自默认工作台，侧边栏高亮正确 |
| U3 | 后台功能页恢复 | 每个角色点击侧边栏功能后刷新 | `role` 和 `page` 恢复正确，不跳回总览 |
| U4 | 关键对象恢复 | 在客户 360、老师学生服务、报告详情等带对象上下文页面刷新 | 当前对象仍可识别，列表选中态和详情一致 |
| U5 | 手动 URL 越权 | 手动把学生 URL 改成管理员页面，把员工 URL 改成老师页面 | 不展示无权限业务数据，返回允许访问的入口或业务化提示 |

## 跨角色业务闭环

| 编号 | 动线 | 操作 | 通过标准 |
| --- | --- | --- | --- |
| B1 | 官网活动报名到顾问承接 | 官网活动页提交报名，顾问进入活动或线索相关入口查看 | 报名对象可查到，人数或线索记录同步变化 |
| B2 | 顾问新建线索到客户 360 | 顾问手动录入线索，进入线索队列和客户 360 | 同一客户资料、阶段、来源一致 |
| B3 | 顾问跟进到管理者增长 | 顾问新增跟进、任务、阶段变更，管理者看增长总览或客户经营报告 | 管理侧指标或记录反映变化，刷新后不丢失 |
| B4 | 员工快捷录入到顾问/管理者 | 员工提交客户快捷录入或状态更新 | 顾问侧可继续跟进，管理侧不出现孤立记录 |
| B5 | 员工日报到管理者日报 | 员工提交日报，管理者进入日报汇总 | 汇总数量、风险、明日动作与日报内容一致 |
| B6 | 学生请假到老师审批 | 学生提交请假，老师审批同意或驳回 | 学生侧状态、老师队列、管理侧摘要同步 |
| B7 | 学生反馈到老师处理 | 学生提交反馈，老师记录处理结果 | 学生侧能看到处理状态和记录，反馈队列数量更新 |
| B8 | 老师成绩录入到学生查询 | 老师录入或修改成绩，学生查看学业/成绩 | 科目、分数、备注一致，刷新后仍存在 |
| B9 | 老师心理辅助预警到管理者周报 | 老师记录心理辅助跟进，管理者查看心理周报 | 只展示辅助识别和跟进语言，不出现诊断表述 |
| B10 | 报告生成到详情查看 | 管理者生成客户经营、日报、心理、投诉类报告 | 报告列表新增记录，详情快照可打开，刷新后仍可查看 |

## 写操作同步检查

每个带提交、审批、处理、更新、生成的按钮都按以下口径验收：

1. 提交前校验必填字段，错误提示是业务语言。
2. 提交中按钮有禁用或进行中反馈。
3. 成功后当前详情、列表、数量指标同步变化。
4. 失败后保留用户输入，错误可理解。
5. 刷新浏览器后服务端状态与提交后页面一致。
6. 切换到相关角色后能看到同一个业务对象的最新状态。

## 后台布局回归视口

| 视口 | 重点检查 | 通过标准 |
| --- | --- | --- |
| 1366 x 768 | 首屏密度、右侧空白、按钮遮挡 | 主操作可见，列表不把页面撑出横向滚动 |
| 1920 x 1080 | 宽屏信息分布 | 内容不飘散，卡片和列表有合理最大宽度 |
| 390 x 844 | 移动端堆叠 | 文案不溢出按钮，卡片按业务顺序单列展示 |

## 普通业务页文案检查

普通业务角色包括顾问、员工、老师、学生、管理者和公开官网。测试时不应看到以下话术：

- 真实 API
- fallback
- seed
- 原型数据
- 后续 V2 / V3
- 后续接入
- 接口调试
- template_rule
- OpenAPI

管理员系统治理页可以展示系统状态、初始化、接口和知识来源相关信息。

## 建议测试顺序

1. 先测入口与 URL 恢复，避免后续动线记录无法复现。
2. 再测跨角色业务闭环，优先确认数据对象是否同步。
3. 接着测写操作同步，覆盖列表、详情、指标和刷新。
4. 最后做三档视口布局回归，再决定是否进入后台视觉统一改版。

## 是否进入视觉/UI 调整的判断

满足以下条件后再开始后台视觉和 UI 精修：

1. U1 到 U5 没有阻断问题。
2. B1 到 B10 至少主链路能跑通，允许存在低优先级文案或局部布局问题。
3. 普通业务页没有明显实现话术。
4. 写操作刷新后不丢状态。

如果上述条件未满足，优先修业务闭环；如果已满足，可以开始后台视觉治理，但每次只按一个角色或一组同类页面推进。
