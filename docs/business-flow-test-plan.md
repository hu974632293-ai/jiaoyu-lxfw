# 业务动线、单功能操作与界面可用性测试计划

## 测试目标

在后台视觉和 UI 精修前，完整确认系统界面功能可用、业务对象可同步、刷新后可恢复、权限边界不泄露、普通业务角色不出现实现话术。测试范围同时覆盖两类问题：

1. 跨角色业务动线：同一个业务对象从创建、处理到管理侧回显是否一致。
2. 单页面功能操作：搜索、筛选、切换、提交、刷新、详情、空状态、错误状态是否可用。

本计划升级为系统功能验收计划。验收结论必须区分四层：

1. 前端界面验收通过：页面、交互、布局、文案和 URL 恢复通过。
2. 后端 API 验收通过：接口行为、权限、错误返回和服务层规则通过。
3. 数据库验收通过：写入、读取、关联、刷新持久化和当前运行库专项检查通过。
4. 系统整体验收通过：前端、后端、数据库、权限和跨角色业务闭环全部通过。

## 测试前准备

1. 启动后端和前端，确认浏览器访问 `http://localhost:5173/`。
2. 如需重置演示数据，只在管理员系统治理入口执行初始化。
3. 每条测试记录至少包含：角色、入口 URL、操作对象名称或编号、预期结果、实际结果、是否刷新验证。
4. 每发现问题先分类：数据不同步、URL 丢失、权限绕过、操作反馈不清楚、布局遮挡、文案不合规、接口错误。
5. 每轮测试完成后运行：
   - `cd frontend && npm.cmd run build`
   - `cd backend && python -m pytest -v`
   - `git diff --check`

## 通过标准

1. 所有角色默认入口、侧边栏入口、总览卡片入口可点击并指向正确页面。
2. 所有写操作成功后，列表、详情、数量指标和相关角色页面同步更新。
3. 刷新浏览器后，当前角色、页面、关键对象 ID 和选中状态不丢失。
4. 搜索、筛选、详情切换、折叠展开、刷新按钮、表单校验都有明确反馈。
5. 无权限页面不展示业务数据。
6. 普通业务页不出现实现话术或阶段规划话术。
7. 三档视口无横向滚动、遮挡、明显大空白、长列表失控堆叠。
8. 后端自动化测试通过，且接口错误返回符合统一结构。
9. 当前运行数据库完成专项检查，写入后的业务对象刷新后仍存在。
10. 5 名验收人各自任务完成，并完成至少 1 次跨组复核。

## 5 人验收分工

分工原则：按业务依赖、验收量和专业关注点分配。每个人都负责一组主任务和一组复核任务，避免只测自己创建的数据而漏掉跨角色同步问题。

| 验收人 | 主验收范围 | 重点测试编号 | 主要产出 | 复核范围 |
| --- | --- | --- | --- | --- |
| 验收人 A：入口与公开侧 | 公开官网、登录、后台外壳、URL 恢复、普通业务页文案、三档视口布局 | U1-U8、P1-P11、S1-S5、布局回归、文案检查 | 公开入口可用性记录、URL 恢复记录、布局截图、文案问题清单 | 复核验收人 D 的管理者报告详情和视觉遮挡问题 |
| 验收人 B：客户增长侧 | 顾问、客户 360、线索、跟进、任务、项目推荐、活动邀约 | C1-C11、K1-K8、B1-B3、B11-B12 | CRM 对象编号、线索到客户 360 的同步记录、活动报名承接记录 | 复核验收人 E 的 API/数据库中 CRM、活动、报告相关记录 |
| 验收人 C：学生服务侧 | 员工、老师、学生三类日常操作，重点看服务对象状态同步 | E1-E9、T1-T10、ST1-ST8、B4-B9 | 请假、反馈、成绩、日报、心理辅助跟进的跨角色回显记录 | 复核验收人 A 的移动端堆叠和老师/员工页面布局问题 |
| 验收人 D：管理治理侧 | 管理者、管理员、运营资源、报告中心、权限绕过和审计 | M1-M8、A1-A8、O1-O9、B10、权限验收 | 管理报表、审计、权限、报告快照、系统治理验收记录 | 复核验收人 B 的顾问任务、活动邀约和报告跳转 |

## 5 人协作规则

1. 每个人先完成自己的主验收范围，再执行复核范围。
2. 每个人至少提交 1 份验收记录，格式见“缺陷记录格式”和“最终验收汇总格式”。
3. 跨角色动线必须由两人配合完成：创建方记录对象编号，处理方或管理方确认回显。
4. 验收人 E 不替其他人点界面，但负责确认关键对象是否通过 API 或数据库持久化。
5. 任一阻断级问题出现时，暂停视觉/UI 精修，优先修功能或数据问题。
6. 中低优先级布局问题可以进入视觉治理清单，但必须有截图和 URL。

## 通用单功能验收口径

每个搜索、筛选、切换、提交、刷新、详情、折叠、跳转都按以下口径验收：

1. 操作前：按钮、输入框、筛选器可见且文案清晰。
2. 操作中：有加载、禁用、选中或进行中状态。
3. 操作后：当前区域、关联列表、详情和数量指标同步变化。
4. 失败时：错误提示是业务语言，保留用户已输入内容。
5. 空数据：有空状态，不出现大白板或布局塌陷。
6. 重复操作：重复点击不会重复创建脏数据或导致页面崩溃。
7. 刷新后：服务端状态与页面显示一致。

## 普通业务页文案检查

普通业务角色包括顾问、员工、老师、学生、管理者和公开官网。测试时不应看到以下话术：

- 真实 API
- fallback
- seed
- 原型数据
- 后续 V2 / V3
- 后续接入
- 接口调试
- template_rule
- OpenAPI

管理员系统治理页可以展示系统状态、初始化、接口和知识来源相关信息。

## 缺陷记录格式

| 字段 | 记录要求 |
| --- | --- |
| 缺陷编号 | 建议按角色缩写编号，例如 `T-001`、`C-002` |
| 角色/页面 | 写清楚入口角色和页面名称 |
| URL | 粘贴完整 URL，尤其是 `role`、`page`、对象 ID |
| 操作步骤 | 控制在 3 到 6 步，可复现 |
| 预期结果 | 业务上应该发生什么 |
| 实际结果 | 实际看到什么，附截图更好 |
| 类型 | 数据、权限、布局、文案、交互、接口、刷新恢复 |
| 严重度 | 阻断 / 高 / 中 / 低 |

## 建议测试顺序

1. 先测入口与 URL 恢复，避免后续动线记录无法复现。
2. 再测公开官网和登录入口，确认外部入口没有内部信息暴露。
3. 按角色测单页面功能：顾问、员工、老师、学生、管理者、管理员。
4. 测运营资源、活动和报告中心。
5. 测跨角色业务闭环。
6. 做三档视口布局回归。
7. 汇总阻断和高优先级缺陷，修复后再进入后台视觉统一改版。

## 是否进入视觉/UI 调整的判断

满足以下条件后再开始后台视觉和 UI 精修：

1. U1 到 U8 没有阻断问题。
2. P、S、C、K、E、T、ST、M、A、O 各组没有阻断级单功能问题。
3. B1 到 B12 至少主链路能跑通，允许存在低优先级文案或局部布局问题。
4. 普通业务页没有明显实现话术。
5. 写操作刷新后不丢状态。

如果上述条件未满足，优先修业务闭环和单功能可用性；如果已满足，可以开始后台视觉治理，但每次只按一个角色或一组同类页面推进。
