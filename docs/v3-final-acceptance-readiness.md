更新日期：2026-06-15

## 1. 目标和边界

本文件用于补齐最终 B1-B12 验收准备，不替代 `docs/business-flow-test-plan.md`、`docs/test-object-claim-table.md` 和 `docs/verification-checklist.md` 的执行记录。

假设：

- 本批只做最终验收准备、证据矩阵和风险汇总，不改真实业务流程。
- 真实 Dify key/app/dataset 不作为本批阻塞项；本批只把它列入上线后配置验收。
- 当前事实以 `AGENTS.md`、`docs/v3-current-consistency-audit.md`、当前代码和测试结果为准。

取舍：

- 优先把已完成批次、自动化验证、人工验收和上线配置边界落到文档和轻量测试。
- 不重跑所有人工链路，不重构业务代码，不新增真实外部账号或密钥。
- SQLite 与 MySQL 验收结果必须分开记录，不能用 SQLite 通过替代 MySQL 通过。

成功标准：

- B1-B12 每条链路都能追踪到角色边界、前端入口、API/对象、自动验证命令、人工验收项、上线配置项和已知风险。
- 证据明确分为：已自动验证、需人工验收、上线后配置项。
- 后续验收人可以按本文件确认准备状态，再到执行清单中记录真实对象 ID、URL、截图和数据库/API 抽查结果。

## 5. 最终验收执行顺序

1. 运行自动门禁：`cd backend && python -m pytest -v`、`cd frontend && npm.cmd run build`、`git diff --check`。
2. 按 `docs/test-object-claim-table.md` 完成 A-E 对象认领，记录实际 ID、名称和 URL。
3. 先验 U1-U9、P1-P11、S1-S7，确认入口、登录、URL 恢复和公开侧边界。
4. 再验 C/K/E/T/ST/M/A/O 单功能，确认搜索、筛选、提交、处理、错误和空状态。
5. 按 B1-B12 跑跨角色闭环，每条至少记录创建方、处理方、管理侧或只读复核方证据。
6. 验收人 E 抽查关键对象 API 或数据库持久化，并写明当前运行库是 SQLite 还是 MySQL。
7. 汇总阻断和高优先级问题；阻断未清零前不进入视觉/UI 精修。
