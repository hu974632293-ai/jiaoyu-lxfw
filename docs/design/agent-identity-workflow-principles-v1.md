# Agent 身份化协作与状态提示设计原则 v1

## 1. 目标

本文件用于收口后续所有 Agent 页面和能力的产品逻辑，避免把 Agent 做成测试引导器、页面说明组件或内部流程展示器。

最终原则：所有 Agent 都必须围绕真实使用者身份提供便利。Agent 只展示该身份在当前业务任务中真正需要知道、确认和执行的内容。

## 2. 核心判断

1. Agent 不是验收脚本，也不是系统内部流程说明。
2. Agent 不应在普通业务页面反复解释“如何使用后台”。后台操作帮助、制度流程、入口说明统一归入“新人指南”或对应帮助场景。
3. Agent 的主要价值是减少输入、理解意图、生成草稿、解释结果、提示风险、引导下一步和确认业务动作。
4. 用户不需要看到内部过程名，例如“识别意图、生成草稿、写入记录、跳转处理”这类工程化流程。
5. 用户需要看到的是业务状态，例如“正在整理日报草稿”“待本人确认”“已保存到日报”“可进入工作台继续处理”。

## 3. 角色 Agent 边界

企业版 Agent 基础使用权只授予内部后台角色进入对应助手的能力；基础公共能力限于组织联系人、新人指南、制度流程和本人工作记录。客户、学生、审批、报告、治理等业务能力必须按角色和对象权限单独授权，不随企业版基础权限自动开放。

| 使用者 | Agent 目标 | 应提供的便利 | 不应承担 |
| --- | --- | --- | --- |
| 游客/潜在客户 | 客服咨询 | 公司、业务、政策、项目、活动、FAQ 和报名引导 | 内部 CRM、日报、学生风险、系统治理 |
| 顾问 | 企业版顾问助手 | 组织查询、新人指南、本人日报、客户资料摘要、画像解释、项目推荐、跟进建议、线索状态流转 | 学生审批、系统治理、全员日报 |
| 员工 | 企业版员工助手 | 组织查询、新人指南、本人日报、受控查询；客户能力仅在明确协作授权时开放 | 管理报表解读、学生心理明细、默认客户查询、系统治理 |
| 老师 | 企业版老师助手 | 组织查询、新人指南、本人日报、请假审批建议、反馈摘要、心理辅助跟进、学业节点查看 | 顾问 CRM、经营管理报表、默认客户查询 |
| 学生 | 学生助手 | 请假/反馈草稿、申请进度、考务节点、生活支持、心理倾诉 | 内部处理队列、老师审批视角 |
| 管理者 | 企业版管理助手 | 组织查询、新人指南、日报汇总、报告生成、指标解释、风险定位、经营摘要 | 一线提交动作、学生自助提交、默认客户明细操作 |
| 管理员 | 企业版治理助手 | 组织查询、新人指南、用户、权限、知识来源、审计和系统状态治理 | 业务角色的日常工作台合集、日常客户跟进 |

### 3.1 权限分层

| 权限层 | 含义 | 默认范围 | 不包含 |
| --- | --- | --- | --- |
| 企业版基础入口 | 进入对应企业版 Agent，并使用公共组织级能力 | 顾问、员工、老师、管理者、管理员 | 游客、学生 |
| 公共组织能力 | 组织联系人、新人指南、制度流程、本人工作记录 | 内部后台角色按自身身份使用 | 客户明细、学生明细、经营报表、系统治理 |
| 角色业务能力 | 客户、学生服务、报告、治理等具体业务动作 | 按角色、对象范围和业务权限授权 | 随企业版基础入口自动开放 |
| 写库确认能力 | Agent 草稿确认后调用业务 service 写入 | 必须经过 token/current user、业务权限、幂等和审计 | 只凭 URL account 或前端入口判断 |

### 3.2 基础能力与角色业务能力

| 能力 | 顾问 | 员工 | 老师 | 管理者 | 管理员 |
| --- | --- | --- | --- | --- | --- |
| 企业版 Agent 基础入口 | 有 | 有 | 有 | 有 | 有 |
| 组织/联系人查询 | 有 | 有 | 有 | 有 | 有 |
| 新人指南/制度流程 | 有 | 有 | 有 | 有 | 有 |
| 我的日报/个人工作记录 | 本人提交 | 本人提交 | 本人提交 | 查看汇总 | 不作为日常入口 |
| 客户查询/客户跟进 | 有 | 仅限授权协作场景 | 默认无 | 汇总和报表 | 治理配置，不做日常跟进 |
| 学生服务对象 | 默认无 | 默认无 | 有 | 汇总和报告 | 治理配置 |
| 报告与经营指标 | 本人或团队相关 | 默认无 | 学生服务相关 | 有 | 配置和审计 |
| 权限/审计/知识源 | 无 | 无 | 无 | 只读或汇总 | 有 |

## 4. 状态提示替代“执行队列”

后续 Agent 页面不默认展示内部“执行队列”。如需显示当前进度，应使用面向业务用户的状态提示。

推荐结构：

| 区域 | 内容 | 示例 |
| --- | --- | --- |
| 当前任务 | 用户正在让 Agent 处理的业务目标 | 日报草稿、查询负责人、客户摘要 |
| 当前状态 | 结果是否生成、是否需要确认、是否已保存 | 整理中、待确认、已保存、需要补充 |
| 下一步 | 用户可以直接执行的动作 | 编辑草稿、确认提交、查看详情、进入工作台 |
| 影响范围 | 必要时说明动作会写入哪里 | 保存到我的日报、进入客户记录、通知老师处理 |

不推荐展示：

```text
识别意图
生成草稿
写入记录
跳转处理
匹配业务场景
等待业务指令
```

这些属于系统内部过程，不应作为交付给客户的主界面文案。

## 5. 弹出卡片和确认机制

弹出卡片不是输入区的一部分，也不是每次点击入口都必须出现。它只在“状态发生变化”或“需要用户确认”时出现。

### 5.1 何时出现

1. Agent 已生成可确认草稿，例如日报、线索、请假、反馈、报告条件。
2. Agent 已完成查询，并返回可跳转对象，例如负责人、客户、学生事项、报告。
3. Agent 识别到风险或缺失信息，需要用户补充。
4. 用户确认提交后，需要反馈“已保存/已提交/已进入处理队列”。

### 5.2 何时不出现

1. 用户只是切换场景，例如点击“日报 / 组织 / 客户 / 指南”。
2. 用户只是查看可选操作入口。
3. Agent 只是在等待输入。
4. 页面已经有清晰的业务详情区，弹出卡片会重复信息。

### 5.3 显示方式

1. 卡片默认轻量，不抢占主聊天空间。
2. 卡片应可关闭、折叠或被新结果替换。
3. 卡片只展示结果摘要和下一步动作，不展示长篇说明。
4. 详细内容通过“查看详情 / 编辑草稿 / 进入工作台”承接。
5. 输入框和主要对话历史必须始终可见或易于回到。

## 6. 企业助手改造方向

当前企业版 Agent 不是统一菜单模板。每个角色进入自己的企业版助手，公共能力和角色业务能力按权限渲染；右侧“执行队列”应调整为业务状态区，而不是内部流程队列。

建议结构：

| 旧区域 | 新区域 |
| --- | --- |
| 执行队列 | 当前任务 |
| 识别意图 | 任务类型 |
| 生成草稿 | 处理状态 |
| 写入记录 | 确认动作 |
| 跳转处理 | 下一步 |

示例文案：

```text
当前任务：日报草稿
状态：待本人确认
下一步：编辑草稿 / 提交日报 / 进入日报页
```

顶部“日报 / 组织 / 客户 / 指南”和主区快捷按钮只负责选择场景或发起任务，不负责直接展示大卡片。

顶部场景入口必须按角色能力生成，不对所有企业版角色固定展示同一组入口。组织和指南属于公共能力；日报、客户、学生、报告、治理等入口按角色业务权限展示。

## 7. 后续实施边界

1. 先改企业版 Agent 权限和入口边界，再继续扩展角色专属能力。
2. 不重做已确认的企业助手整体视觉，只调整不合理的流程文案、状态区和结果卡出现逻辑。
3. 后续学生、老师、顾问、管理者 Agent 复用同一结构化动作原则，但按角色任务、权限和对象范围重写文案、入口和状态。
4. 新人指南承担“后台怎么用、制度怎么走、入口在哪里”的帮助职责。
5. 普通业务角色页面继续避免实现话术，管理员治理页可展示系统状态和技术治理信息。
6. Agent 写入动作必须和工作台手动操作走同一套业务 service、权限、幂等和审计，不允许只凭前端角色或 URL 参数放行。

## 8. 验收标准

1. 每个 Agent 页面能说明它服务哪个真实身份和业务任务。
2. 页面不再把内部执行步骤当作主界面文案。
3. 点击场景切换不直接弹出大结果卡。
4. 只有生成结果、需要确认、状态变化或风险提示时才出现结果/状态卡。
5. 结果/状态卡不遮挡主聊天区和输入框。
6. 卡片提供明确下一步动作，并能进入对应业务页面或确认提交。
7. 企业版基础入口不会自动开放客户、学生、报告或治理等业务能力。
8. 后续所有 Agent 设计和实现都应引用本文件作为约束。
