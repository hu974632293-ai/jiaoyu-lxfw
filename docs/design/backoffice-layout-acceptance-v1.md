# 后台布局验收标准 v1

## 1. 目标

本标准用于约束后台业务页的布局密度、对齐和验收方式。它不替代 PRD、原型结构或业务动线计划，只定义“页面是否达到可交付后台界面”的视觉和结构底线。

后台页面通过功能验证后，仍必须通过本标准。若出现大块无业务含义留白、列线明显不齐或固定模板造成的空列，该页面视为未完成。

## 2. 布局原则

1. 后台优先服务高频业务处理，页面应紧凑、可扫描、可连续操作。
2. 页面布局必须由业务内容决定，不允许为了套模板保留空列、空面板或大面积背景。
3. 单一业务块必须全宽承接，不能只占左半屏。
4. 双列或三列布局必须保证每一列都有真实业务内容：列表、详情、操作、历史、待办或状态反馈。
5. 内容量不足时，优先采用跨列、合并区块、自动填充网格或固定滚动区，而不是留下裸露空白。
6. 同一页面的卡片边线、标题基线、列宽、间距和按钮区域必须对齐。
7. 角色后台只能展示当前角色可执行的业务语言，不用实现话术填补空间。

## 3. 允许的布局模式

### 3.1 单列全宽

适用：权限列表、审计日志、系统状态、单一表格、单一配置页。

验收要求：

- 主面板宽度占满内容区。
- 面板内部使用自动填充网格或表格承接内容。
- 不出现“左侧有内容、右侧空白”的半屏布局。

### 3.2 列表详情

适用：客户 360、请假审批、反馈工单、用户管理、任务处理。

验收要求：

- 左侧为列表或队列，右侧为选中对象详情、操作和历史。
- 没有选中对象时，右侧必须显示明确空状态或最近记录，不保留空白面板。
- 列表高度应可滚动，不能因为列表过长把详情挤到不可见区域。

### 3.3 工作流三段式

适用：创建、处理、状态回显同时存在的业务页。

验收要求：

- 至少包含队列/输入、详情/表单、历史/反馈三个业务承接区。
- 三段之间列宽和上下边线对齐。
- 小屏下按业务顺序堆叠，不改变操作闭环。

### 3.4 自动填充网格

适用：角色卡片、权限点、成绩记录、活动卡片、通知卡片。

验收要求：

- 使用 `auto-fit` 或等效策略填满可用宽度。
- 卡片最小宽度必须有明确下限，避免文字挤压。
- 卡片数量不足时自动扩展，不保留固定空列。

## 4. 失败条件

出现以下任一情况，视为布局验收失败：

1. 一个业务面板只占内容区左半屏，右侧没有真实业务内容。
2. 左右两列高度差过大，短列下方出现连续大块空白。
3. 页面内卡片左边缘、标题基线或主要按钮区域明显错位。
4. 同一页面混用多个无关 `gap`、`padding` 或列宽比例，导致视觉节奏不统一。
5. 列表、详情、操作、历史任一关键承接区缺失，只剩静态卡片。
6. 使用“暂无”“后续”“原型”“真实 API”“fallback”等实现话术填补普通业务角色页面空间。
7. 业务页面在刷新后能进入当前路由，但当前对象或业务位置丢失。

## 5. 代码约束

1. 新增后台页面应优先复用统一布局原语，不在页面内随手新增固定双列或三列模板。
2. 固定列布局只能用于确有多列业务内容的页面。
3. 单面板状态必须显式使用全宽布局。
4. 卡片网格优先使用自动填充策略。
5. 页面级 `min-height` 不能用于制造视觉填充；只有列表、详情、历史等可滚动业务区可以设置明确高度。
6. 任何布局修复不得顺手改动无关业务逻辑、接口字段或角色权限。

## 6. 浏览器验收

每批前端布局调整至少检查以下视口：

- 1365 x 768：验证普通笔记本下是否出现空列、遮挡和错位。
- 1440 x 900：验证常见桌面下卡片边线和主要列宽。
- 1920 x 1080：验证宽屏下是否出现过度拉伸或大面积留白。

每个视口至少检查：

1. 左侧导航和主内容边界是否对齐。
2. 页面主面板是否占用合理宽度。
3. 列表、详情、操作、历史是否形成闭环。
4. 文本是否溢出、重叠或被按钮遮挡。
5. 滚动条是否出现在业务列表而不是制造整页大空白。

## 7. 当前样板页

本标准优先用于修正以下已暴露问题的页面：

1. 学生端生活支持页：避免左侧对话区短、右侧记录栈过长导致中部大空白。
2. 管理员角色页：单独查看角色时不得只占左半屏。
3. 管理员权限页：权限点列表必须全宽自动填充，不保留右侧空列。

后续所有后台返工和新增页面均以本标准作为布局验收依据。
