# 前端视觉基线 v1

## 1. 设计结论

当前前端采用“B 清爽数据增长”作为系统底盘，同时吸收 A 和 C 的优点：

1. 官网首页：采用 A 的教育咨询信任感，叠加 B 的项目转化结构，并允许局部使用 C 的背景图、层次感和智能咨询氛围。
2. 内部人员后台：采用 B 的 SaaS 高效结构，优先支持表格、筛选、看板、抽屉、任务、审批和报告。
3. 学生端：采用 B 的清爽结构，但做柔化处理，减少表格感，增加服务入口、我的事项、问答支持和亲和插图。
4. AI/智能模块：可局部使用 C 的科技感，例如客服 Agent、画像研判、智能推荐、报告洞察，不作为全站通用底色。

## 2. 角色视觉分层

| 角色/入口 | 视觉目标 | 页面特征 |
| --- | --- | --- |
| 游客/潜在客户官网 | 建立信任、解释服务、促成咨询 | 真实教育场景图片、品牌首屏、项目/活动 CTA、客服 Agent 入口 |
| 顾问 | 高效客户增长 | 新建线索、研判、推荐、跟进、客户 360；信息密度中高 |
| 员工 | 快速内部办理 | 客户快捷录入/查询、日报、组织查询、新人指南；操作入口清晰 |
| 老师 | 待办和风险处理 | 请假审批、反馈工单、心理辅助预警、学业进度；任务优先 |
| 管理者 | 经营和风险观察 | 报告、趋势、指标、异常；图表和摘要优先 |
| 管理员 | 系统治理 | 用户、角色、权限、审计、通知、知识来源；表格和配置优先 |
| 学生 | 轻量自助服务 | 请假、反馈、进度、考务、生活支持、心理倾诉；少表格、少后台感 |

## 3. 字体原则

字体可在实现和精修阶段继续调整。TypeScript 不决定字体，字体由 CSS 的 `font-family`、`font-size`、`font-weight`、`line-height` 和实际引入的字体资源决定。

1. 官网标题可以更有品牌感，字号更大、字重更强、行距更紧。
2. 官网正文以易读和咨询感为主，避免过长说明。
3. 内部后台使用清晰克制的字体层级，适合长期操作和数据阅读。
4. 学生端字体略大、行距更松，降低压迫感。
5. 表格、表单和按钮使用稳定字号，避免因视觉装饰影响可读性。

## 4. 图片和插图原则

当前风格确认不限制后续加入插图、背景图或真实照片。图片使用位置按角色和任务区分：

1. 官网首页、项目页、活动页可以使用真实教育场景图、校园图、海外城市图、学生/顾问沟通图和品牌插图。
2. 学生端可以使用柔和插图、服务图标、生活支持场景图。
3. 内部人员后台少用大背景图，避免影响表格、审批、任务处理效率。
4. AI/智能模块可以使用局部科技感视觉，但不得让所有后台页面变成深色大屏。
5. 图片应服务业务判断和信任建立，不做无意义装饰。
6. 登录页背景图和整体视觉方向必须衔接公开官网风格，避免像另一个独立系统入口。
7. 后续使用外链图片、背景图、图标、字体或其他静态素材时，必须优先保存到项目内再引用；图片默认放在 `frontend/src/assets/images/`，避免外链失效影响演示和上线。

## 5. 实现边界

1. 官网不得展示 CRM 客户、客户 360、员工日报、学生心理预警明细、审计日志、权限矩阵、OpenAPI、seed、接口健康等内部信息。
2. 内部人员后台和学生端必须明显区分，不能把学生端做成员工后台换皮。
3. 后续可在内部后台引入 Ant Design；官网以自定义视觉为主，学生端可复用少量基础控件但需要柔化。
4. 选定方向后，正式 React 实现应优先改造官网首页、顾问后台和学生服务台三个代表性入口，再扩展到其他角色。
5. 进入视觉实现前，先做布局治理审计，只记录问题清单、风险等级和建议处理阶段；不要在审计阶段改菜单、路由、表单字段、接口调用或状态流转。
6. 管理员视觉重点是系统治理；测试账号的“演示视角”是验收工具，不作为生产角色视觉层级。
