# 顾问客户增长 Agent 黄金链路 v1

## 目标

本链路只验证一件事：顾问面对一个真实客户时，能否用自然语言让系统读取客户上下文，生成可确认业务动作，并在顾问确认后写入 CRM，刷新后能在客户 360、任务和阶段记录中追踪。

## 边界

- 工作台负责确定性业务：客户列表、客户 360、跟进记录、任务、阶段流转、报表回显。
- Agent 负责辅助：理解自然语言、读取当前客户上下文、生成解释、跟进草稿、任务草稿和阶段建议。
- Agent 不直接写库；所有写操作必须先返回可确认草稿，再由顾问确认后调用正式业务 service。
- Dify/LLM 负责理解和表达；后端编排层负责角色上下文、权限、动作分派、确认、审计和幂等。
- 本地 fallback 只能作为降级，不作为真实智能命中结论。

## 最小闭环

1. 顾问在客户增长工作台或顾问 Agent 中选中一个真实客户。
2. 顾问输入自然语言，例如“帮我给这个客户准备今天的跟进，并安排三天后回访”。
3. Agent 读取客户名称、来源、当前阶段、资料背景、最近跟进、任务和阶段历史。
4. Agent 返回一组可确认动作：
   - 新增跟进记录草稿；
   - 创建 CRM 任务草稿；
   - 可选的阶段更新建议。
5. 顾问确认后，后端复用 CRM 正式 service 写入跟进、任务和阶段历史。
6. 刷新客户 360 后，时间线能看到新增跟进、创建任务和阶段流转。

## MVP 成功标准

- 不是只要求“能发消息、能返回文案”。
- 必须识别顾问跟进类意图。
- 必须绑定真实 `lead_id`，不存在或无权限时返回业务错误。
- 必须读取真实客户上下文生成草稿。
- 必须返回 `requires_confirmation=true` 和结构化 `pending_actions`。
- 确认接口必须调用 CRM 正式 service，不新造绕路写库。
- 确认后必须返回写入结果，且客户时间线可追踪。
- 普通顾问页面不展示 “fallback / seed / OpenAPI / V2/V3” 等实现话术。

## 本轮不做

- 不扩展老师、学生、管理者和管理员 Agent。
- 不继续微调角色助手视觉。
- 不配置真实 Dify key/app/dataset。
- 不让 Agent 直接审批、直接改权限或绕过 CRM 权限。
