# 后台业务动线重构方案

## 1. 目标

本方案用于修正当前后台“总览卡片多、功能混杂、右侧空白、点击后仍是同一大页面”的问题。

目标不是替换 UI 风格，而是保留当前后台视觉基准，重构业务动线和页面承载方式。

最终入口结构保持：

```text
公开官网门户 -> 登录入口 -> 按角色进入后台总览 -> 点击具体功能 -> 独立业务页面
```

## 2. 已确认的设计边界

| 项 | 决策 |
| --- | --- |
| 视觉基准 | 以当前后台截图为准，保留蓝白、浅色背景、圆角卡片、图标、状态标签和左侧导航风格 |
| 不采用 | 不采用生成图中的窄侧栏、朴素线框风、通用模板后台皮肤 |
| 左侧导航 | 保留当前图标 + 标题 + 描述 + 选中态卡片的视觉，不改成细窄二级菜单 |
| 右侧内容 | 按独立业务页重构，必须用满可用宽度，避免大面积空白 |
| 工作台总览 | 只做指标、待办、最近记录和功能入口，不承载所有业务细节 |
| 功能页 | 每个业务入口点击后进入独立业务页，不再把多条动线混在一个角色页面中 |

## 3. 统一页面模型

每个独立业务页按业务需要组合以下区域：

| 区域 | 作用 | 验收要求 |
| --- | --- | --- |
| 页面标题 | 告诉用户当前正在处理什么任务 | 标题是业务语言，不写阶段、真实 API、fallback、seed 等实现话术 |
| 指标摘要 | 显示本页相关数量、风险、待办 | 只显示与当前功能有关的指标 |
| 搜索/筛选 | 缩小列表范围 | 改变列表、数量或空状态 |
| 列表/队列 | 展示可处理对象 | 点击后选中对象或进入详情 |
| 详情区 | 展示当前对象完整信息 | 包含状态、发起人、时间、业务字段 |
| 操作区 | 提供创建、编辑、审批、处理、归档等动作 | 写操作必须有按钮态、结果反馈和失败保留 |
| 历史记录 | 展示审批、处理、跟进、审计或时间线 | 状态流转必须留痕 |
| 空/错状态 | 没数据或接口失败时展示 | 给业务原因或恢复动作，不暴露内部技术状态 |

## 5. 视觉和布局规则

1. 视觉基准以当前后台为准：浅色背景、圆角白卡、蓝色主按钮、状态标签、左侧大导航项。
2. 不改成窄侧栏、灰白线框或通用模板后台。
3. 页面最大宽度和网格必须充分利用当前浏览器宽度。
4. 功能少的页面使用紧凑两栏或三栏；功能多的页面使用分区布局。
5. 不允许右侧大面积空白。
6. 不允许窄列挤压导致文字一字一行。
7. 业务页面不写阶段说明、实现状态或技术 fallback 话术。
8. 学生端可以比内部后台更柔和，但不能变成另一个无操作能力的展示页。

## 6. URL 和状态保持

每个独立业务页必须具备可恢复位置：

| 状态 | 方式 |
| --- | --- |
| 当前角色 | query 或明确持久化 |
| 当前页面 | `page` query 或路由段 |
| 当前业务对象 | `leadId`、`studentId`、`leaveId`、`ticketId`、`reportId` 等 query |
| 当前筛选 | 可放 query 或本地持久化，但刷新后不能丢失关键上下文 |

示例：

```text
?mode=backoffice&role=teacher&page=teacherLeaveApproval&leaveId=12
?mode=backoffice&role=student&page=studentLeaveApplication&leaveId=12
?mode=backoffice&role=consultant&page=customer360&leadId=3
```

## 7. 实施原则

1. 先整体拆动线，再分批改代码。
2. 不一次性重写全站 UI。
3. 每批只改本批页面和必要 API。
4. 不删除已有已验证能力，只移动到更合适的角色和页面。
5. 每批完成后运行后端测试和前端构建。
6. 每批单独中文 commit。
7. 进入整体视觉方案前，先做布局治理审计：只记录错位、留白、遮挡、列表失控、移动端堆叠等问题，不直接调整结构。
8. 布局治理审计不改变测试计划；只有实际改动菜单、路由、表单字段、按钮业务行为、接口调用或状态流转时，才需要同步更新验收计划。

## 8. 视觉前布局治理分级

| 等级 | 示例 | 处理顺序 |
| --- | --- | --- |
| L1 视觉阶段可修 | 间距不统一、卡片高度不齐、按钮层级不清、局部留白 | 进入视觉实现阶段处理，不影响功能验收 |
| L2 需要单独布局改造 | 页面信息结构混杂、列表和详情承载关系不清、移动端无法完成关键操作 | 先记录并确认范围，再单独实现和更新测试计划 |
| L3 功能或权限问题 | 菜单权限错误、按钮行为错误、URL 越权、表单字段或接口调用需要变化 | 退出视觉流程，回到功能修复和验收计划同步 |

## 9. 第一批样板范围

第一批只建立学生服务闭环样板：

1. 学生请假申请。
2. 老师请假审批。
3. 学生反馈提交。
4. 老师反馈处理。

第一批验收通过后，再复制同一页面模型到成绩、学业进度、CRM、日报、报告和治理页。
