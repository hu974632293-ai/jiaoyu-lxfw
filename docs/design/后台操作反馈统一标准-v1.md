# 后台操作反馈统一标准 v1

## 1. 背景与目标

用户反馈集中在两个问题：

1. 刷新、新建、保存、生成、审批等按钮缺少显眼的成功、失败、进行中信号。
2. 部分功能卡片和按钮语义不清，点击后不知道发生了什么，也不知道下一步该做什么。

这不是单个页面的样式问题，而是后台缺少统一的操作反馈系统。本文用于约束后续前端改造，先统一标准，再分批实施。

目标：

1. 任何会请求接口、改变数据、生成内容、切换业务上下文的操作，都必须有可见反馈。
2. 写操作必须让用户明确知道：正在处理什么、是否成功、成功后产生了什么结果、失败后如何恢复。
3. 功能入口、统计卡、执行按钮必须在视觉和交互上区分，避免用户误以为所有卡片都可执行。
4. 后台页面以生产力工具为目标，反馈要靠近操作区域，并能定位到具体业务对象。

非目标：

1. 不在本标准中重做业务流程。
2. 不一次性改造全部页面代码。
3. 不把所有提示都做成全局 toast，避免反馈离业务上下文太远。

## 2. 适用范围

适用于登录后的业务后台和学生服务台，包括：

1. 顾问客户增长与客户 360。
2. 员工工作台与企业助手。
3. 老师学生服务工作台。
4. 学生服务台。
5. 管理者经营后台与报告中心。
6. 项目、活动、知识库和系统治理页面。

公开官网只遵循同一基础原则，但不纳入本轮后台操作反馈改造。

## 3. 操作类型与最低反馈要求

| 操作类型 | 示例 | 最低反馈 | 成功后必须定位的结果 |
| --- | --- | --- | --- |
| 只读刷新 | 刷新队列、刷新活动、刷新学生待办 | 按钮 loading、区域状态、最后更新时间 | 刷新的列表或指标 |
| 创建/保存 | 新建线索、保存项目、保存活动、提交日报 | 按钮 loading、表单禁用、成功提示、失败保留输入 | 新记录、被更新记录或详情入口 |
| 状态流转 | 审批请假、处理反馈、完成任务、更新客户阶段 | 按钮 loading、当前对象状态变化、时间线或审计提示 | 新状态、处理人、处理时间 |
| AI/报告生成 | 画像研判、项目推荐、生成报告、知识库问答 | 生成中状态、fallback 标识、结果摘要 | 研判结果、推荐项目、报告详情或回答内容 |
| 导航/展开/筛选 | 切换面板、展开侧栏、搜索筛选 | active 状态、标题或结果数量变化 | 当前视图或筛选结果 |
| 高风险治理动作 | 写入审计、初始化基础数据、同步知识来源 | 明确操作对象、二次确认或强状态提示、失败恢复路径 | 审计记录、任务记录或系统状态 |

规则：

1. 导航类操作不需要成功 toast，但必须有 active 状态和页面标题变化。
2. 写操作不能只改一个远处的状态胶囊，必须在按钮附近或当前业务区域显示结果。
3. 失败后必须保留用户输入，并提供重试入口或明确说明当前不可恢复原因。
4. AI/fallback 场景必须区分“真实成功”“使用 fallback”“接口失败”，不能只显示“完成”。

## 4. 反馈层级

后台操作反馈按五层设计，后续组件化时按操作风险组合使用。

| 层级 | 位置 | 适用场景 | 要求 |
| --- | --- | --- | --- |
| L1 按钮内反馈 | 当前按钮 | 所有异步操作 | loading 文案或图标、禁用重复点击 |
| L2 区域内反馈 | 表单、列表、详情面板上方 | 创建、保存、生成、审批 | 说明对象、结果和下一步 |
| L3 全局轻提示 | 页面顶部或统一 toast 区 | 跨区域成功、系统级失败 | 不替代区域反馈，只做补充 |
| L4 结果定位 | 列表选中、高亮、跳转、抽屉打开 | 新建、生成、状态流转 | 用户能看到刚产生或刚改变的对象 |
| L5 审计/时间线 | 客户时间线、学生处理记录、系统审计 | 关键写操作 | 显示操作已沉淀，不只是前端提示 |

最低组合：

1. 只读刷新：L1 + L2。
2. 创建/保存：L1 + L2 + L4。
3. 状态流转：L1 + L2 + L4 + L5。
4. AI/报告生成：L1 + L2 + L4，关键报告增加 L5。
5. 高风险治理动作：L1 + L2 + L3 + L5，必要时增加确认。

## 5. 按钮与卡片语义规则

按钮命名采用“动词 + 对象 + 结果”的结构。

推荐：

1. “刷新客户队列”，不要只写“刷新”。
2. “保存线索并进入研判”，不要只写“保存”。
3. “审批当前请假”，不要只写“处理”。
4. “生成客户经营报告”，不要只写“生成”。
5. “记录知识同步任务”，不要只写“同步”。

限制：

1. 避免孤立使用“同步”“处理”“推荐”“执行”“提交”等模糊词。
2. 同一页面内同名按钮如果作用对象不同，必须补足对象名。
3. 入口卡只做导航时，样式上应像入口，不应像立即执行按钮。
4. 统计卡默认不可点击；如果可点击，必须有箭头、hover、title 或辅助文案说明去向。
5. 执行按钮要放在明确的操作区，不混在静态指标卡中。

## 6. 状态文案标准

状态文案必须回答三个问题：对象、结果、下一步。

成功文案示例：

1. “线索已创建：王同学。已选中新线索，可继续画像研判。”
2. “请假已审批为：已通过。处理记录已写入学生服务时间线。”
3. “客户经营报告已生成。已打开最新报告详情。”
4. “知识同步任务已记录。当前阶段不执行真实 Dify 上传。”

失败文案示例：

1. “线索创建失败：接口不可用。表单内容已保留，可重试。”
2. “报告生成失败：缺少报告类型。请选择类型后再生成。”
3. “知识库调用失败：Dify 未配置。已保留问题，可稍后重试。”

避免：

1. “操作成功”。
2. “完成”。
3. “请求失败”。
4. “处理中”长期停留且没有可恢复路径。

## 7. 页面问题矩阵

| 页面 | 典型问题 | 改造重点 |
| --- | --- | --- |
| 客户增长 | 刷新、新建、同步、推荐、处理按钮混杂；成功提示离结果对象远 | 新建后选中新线索，研判后打开推荐面板，刷新显示列表时间 |
| 客户 360 | 状态流转、跟进、任务创建和完成需要更明确的时间线反馈 | 每个写操作绑定客户时间线和当前任务状态 |
| 员工工作台 | “同步跟进状态”“生成结构化摘要”等入口语义偏抽象 | 区分助手指令、日报提交、只读查询和客户状态更新 |
| 老师学生服务工作台 | 请假审批、反馈处理、心理预警按钮结果不够可感知 | 当前学生或当前工单要被选中、高亮，并显示处理记录 |
| 学生服务台 | 提交请假、反馈、生活支持对话的结果类型容易混淆 | 按事项类型展示提交结果、工单状态和下一步 |
| 管理者后台 | 生成报告和查看详情反馈弱，列表点击像静态卡片 | 生成后打开报告详情，列表项可点击状态更清楚 |
| 报告中心 | 报告生成缺少生成中、成功定位、失败保留条件 | 生成后选中最新报告并显示摘要 |
| 项目维护 | 保存、新建、推荐规则显示混在一起 | 保存后选中项目，推荐结果与保存状态分离 |
| 活动管理 | 活动保存、报名、签到、刷新名单反馈分散 | 报名或签到后定位名单记录，并显示审计提示 |
| 知识库 | 问答、来源创建、同步任务状态容易混为一谈 | 三类操作分区反馈，fallback 状态清晰展示 |
| 系统治理 | 刷新、审计写入、基础数据初始化风险等级不同 | 高风险动作强化确认、结果写入审计或任务记录 |

## 8. 分批实施顺序

### D1 通用反馈基础

目标：

1. 建立统一的操作状态模型。
2. 补齐按钮 loading、disabled、success、error 的基础样式。
3. 建立区域内反馈和结果高亮的基础 CSS。

建议产物：

1. 通用操作状态类型。
2. 可复用的反馈条或反馈区域组件。
3. 可复用的异步操作包装逻辑。

验收：

1. 至少一个高频页面接入新反馈基础。
2. 前端 build 通过。

### D2 客户增长与客户 360

原因：这是顾问最高频路径，也是新建、研判、推荐、跟进、任务和状态流转最集中的区域。

验收：

1. 新建线索后自动选中或高亮新线索。
2. 画像研判后定位推荐结果。
3. 客户 360 的跟进、任务、状态流转都能看到时间线或审计结果。

### D3 老师与学生服务

原因：请假、反馈、心理辅助预警属于状态流转和服务闭环，必须让用户知道“谁被处理了、处理成什么状态”。

验收：

1. 老师审批和反馈处理有明确当前对象、状态变化和处理记录。
2. 学生提交事项后能看到服务台回复、工单状态和下一步。
3. 心理辅助提示继续保留“辅助识别，不替代诊断”的安全边界。

### D4 报告、项目、活动、知识库、治理

原因：这些页面写操作分散，但都涉及保存、生成、同步或系统状态，适合在 D1-D3 稳定后统一补齐。

验收：

1. 报告生成后打开最新报告详情。
2. 项目和活动保存后定位被保存对象。
3. 活动报名、签到、知识同步、审计写入都有对象级反馈。
4. 治理动作按风险等级显示提示，不再像普通刷新按钮。

## 9. 验收标准

后续每个页面改造完成时必须满足：

1. 关键按钮具备 pending、success、error 三类状态。
2. 异步操作期间禁止重复点击。
3. 成功文案包含业务对象和下一步。
4. 失败文案包含原因和恢复方式。
5. 创建、保存、生成、状态流转后能定位到结果对象。
6. 关键写操作能看到时间线、审计、任务或列表状态变化。
7. 入口卡、统计卡、执行按钮在视觉和交互上可区分。
8. 前端 `npm.cmd run build` 通过。

## 10. 实施边界

1. D1 只做通用反馈能力和少量样板接入，不批量重写所有页面。
2. D2-D4 每批独立验证、独立中文 commit。
3. 不改变后端 API 合约，除非某个结果定位必须补充字段；如需补字段，单独讨论。
4. 不把公开官网改成后台反馈系统。
5. 不删除既有 fallback 逻辑，只让 fallback 状态更清晰。
