# 后台角色权限、视觉收口与系统验收推进计划

> 工作区：`D:\00_Project\jiaoyu_lxfw`  
> 用法：每个大阶段建议开一个新线程执行。复制对应阶段的提示词即可。

## 总体顺序

```text
账号登录与角色权限结构确认
→ 角色结构实现与基础验证
→ 同步更新验收计划和活跃文档
→ 视觉前布局治理审计
→ 登录页与后台整体视觉收口
→ 正式系统功能验收
```

当前不建议直接进入登录页和后台视觉精修。原因是登录页、侧边栏、顶部角色标识、默认工作台、URL 越权和权限边界都会受到角色结构影响。如果先做视觉，再改角色结构，登录页和后台外壳大概率返工。

## 目标角色模型

```text
管理员 = 系统治理后台
测试账号 = 全权限验收账号 + 可切换演示视角
管理者 = 管理者工作台 + 员工工作台
老师 = 老师工作台 + 员工工作台
顾问 = 顾问工作台 + 员工工作台
员工 = 员工工作台
学生 = 学生服务台
```

说明：

- 员工工作台是内部通用能力包，例如日报、组织架构、新人指南、受控客户查询。
- 管理者、老师、顾问本质上也是内部员工，所以拥有员工工作台合理。
- 学生不是内部员工，只能访问学生服务台。
- 管理员负责系统治理，不承担全部业务角色的日常入口合集。
- 测试账号用于验收和演示，保留全权限，并通过右上角“演示视角”切换不同角色的菜单和默认入口；上线前移除或关闭。

## 影响程度

| 项目 | 影响程度 | 说明 |
| --- | --- | --- |
| 现有业务功能 | 中 | 业务本身大多不变，但入口和权限预期会变 |
| 测试计划 | 中高 | 必须补账号登录、测试账号演示视角、复合角色、越权和侧边栏可见性测试 |
| 登录页视觉 | 高 | 登录方式变化会直接影响页面结构，必须等角色登录结构稳定后再精修 |
| 后台整体视觉 | 中 | 侧边栏和顶部角色结构稳定后再做更稳 |
| 后端/数据库 | 中到高 | 取决于是否做真实账号、角色绑定和接口级权限校验 |

## 全局规则

1. 使用简体中文沟通。
2. 先阅读并遵守 `AGENTS.md`。
3. 修改前必须说明：假设、取舍、成功标准。
4. 官网界面已满意，除非用户明确要求，不要调整官网首页、官网栏目、公开内容和官网业务动线。
5. 用户是小白，每个阶段先说明会调用哪些 skill / 工具，以及它们分别做什么。
6. 如果用户选择的方案会明显影响测试，例如改路由、菜单结构、表单字段、按钮行为、权限入口、接口调用、数据状态流，必须主动提醒风险。
7. 普通业务页不要出现“真实 API / fallback / seed / 原型数据 / OpenAPI / V2/V3”等实现话术。
8. 纯文档改动不要跑前后端测试；只做 `git diff --check`，然后中文 commit。
9. 前端视觉或前端逻辑改动后运行：
   - `cd frontend && npm.cmd run build`
   - `git diff --check`
10. 改后端或权限接口后运行：
   - `cd backend && python -m pytest -v`
   - `cd frontend && npm.cmd run build`
   - `git diff --check`
11. 每个阶段完成后创建中文 Git commit。
