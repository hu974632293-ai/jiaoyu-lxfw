# 项目开发规则

## 1. 当前阶段

1. 当前已进入第二期，目标是从“客户增长闭环 demo”升级为“完整教育服务业务系统”。
2. 一期客户增长闭环能力必须保持可用，不得破坏已完成的画像研判、Dify 问答、CRM、活动报名、报告生成和 OpenAPI 分组。
3. `教育服务/` 是项目资料目录，只读取需求和业务资料，不纳入 Git 管理，不在提交中加入该目录内容。
4. 二期产品需求以 `docs/prd/教育服务业务系统PRD-v1.md` 为准，原型结构以 `docs/prd/教育服务业务系统原型结构-v1.md` 为准。
5. 二期实施计划以 `docs/superpowers/plans/2026-06-10-education-service-phase2.md` 为准；实施计划必须服务 PRD，不得用工程任务反向扩大产品范围。

## 2. 架构边界

1. 后端采用 FastAPI + SQLAlchemy，继续按 `models/schemas/services/api` 分层。
2. API 层只做参数接收、依赖注入和服务调用；业务逻辑放在 `services`。
3. 数据访问优先通过 SQLAlchemy model 和 service 函数完成，不在路由中堆复杂查询。
4. 所有 API 统一返回 `{ code, msg, data }`，错误也使用同一结构。
5. 前端采用 React/Vite/TypeScript，定位从“技术演示工作台”升级为“教育服务运营工作台”。
6. 前端请求统一经过 `frontend/src/api/client.ts`，不要在页面中重复封装响应结构。
7. 知识库问答统一调用 Dify；未配置 Dify 时必须提供明确 fallback，不阻断 CRM、画像、活动、报告等主业务。
8. 客户画像、报告、心理预警、推荐等 AI 能力必须保留规则或模板兜底，真实大模型只作为增强。

## 3. 二期开发范围

1. 企业助手：客户自然语言录入/查询/状态更新、口述日报、日报汇总、组织架构、新人指南、受控 NL2SQL。
2. 学生助手：请假申请与审批、心理关怀与预警、售后反馈、学业考务、申请进度、生活支持、二次转化推荐。
3. 完整 CRM：线索详情、跟进记录、待办任务、阶段流转、负责人分配、转化和流失归因。
4. 项目/课程管理：项目维护、标签、适合人群、费用周期、招生条件、推荐规则。
5. 活动运营：活动创建、发布、报名、名单、签到，支持线索和学生两类主体。
6. 报告中心：全域客户经营分析、员工日报日/周汇总、学生心理健康周报、投诉处理周报。
7. 权限/角色：用户、角色、权限点、用户角色绑定、关键操作审计。
8. Dify 增强：知识来源、同步任务记录、按场景问答日志和 fallback 状态。

## 4. 数据库原则

1. 当前可继续使用 SQLite，但表设计和字段命名应兼容后续 MySQL。
2. 新增二期能力优先新增表，避免无迁移工具时直接破坏一期已有表。
3. 如必须修改已有表字段，先补迁移策略或明确兼容方案，再改代码。
4. 关键业务表需保留 `created_at`，有状态流转的表需保留状态字段；涉及操作追踪的能力需写入审计或历史记录。

## 5. AI 与安全边界

1. NL2SQL 只能做受控白名单查询，不允许直接执行任意写 SQL。
2. 大模型输出不得直接写入关键业务表；必须先经过 service 层规则校验或意图映射。
3. 学生心理预警属于高风险场景，只做辅助识别和记录，页面文案不得替代专业心理诊断。
4. 权限前端展示只用于体验，后端关键接口必须预留权限校验入口。

## 6. 执行规则

1. 代码改动前先明确假设、取舍和成功标准；不清楚时先提问。
2. 简单优先，但二期可以按完整需求扩展；每阶段仍需形成可运行的纵向切片。
3. 精准改动，只修改能直接追溯到当前阶段目标的内容，不做无关重构。
4. 大功能实现前先确认 PRD 条目、原型结构和阶段验收标准；如果 PRD 未覆盖，先补 PRD 或明确本阶段不做。
5. 每完成一个阶段必须运行对应验证命令。
6. 当前工作区是 Git 仓库，每次代码改动完成后必须自动创建中文 commit。
7. 提交前先检查 `git status`，只 stage 当前阶段相关文件。
8. 环境级 Git 警告 `C:\Users\97463/.config/git/ignore` 读取失败不影响本项目提交。
