# 项目开发规则

## 1. 当前产品基线

1. 当前最终目标是从零落实一个完整教育服务业务系统，而不是继续按“一期/二期”拆入口。
2. 产品范围以 `教育服务/客户需求表.xlsx` 为最高业务需求来源，并结合 `教育服务/` 下公司信息、公司业务、留学政策、用户研判规则和 SQL 草稿落地。
3. 项目文档以 `docs/prd/教育服务业务系统PRD-v1.md`、`docs/prd/教育服务业务系统原型结构-v1.md` 和最新最终实现计划为准。
4. `教育服务/` 是项目资料目录，只读取需求和业务资料，不纳入 Git 管理，不在提交中加入该目录内容。
5. 已完成的客户增长闭环能力必须保留：客户资料输入、画像研判、Dify 问答、CRM、活动报名、报告生成和 OpenAPI 分组。
6. 后续新增、移动或隐藏任何功能前，必须先判断它服务哪个真实角色、哪个业务任务、哪个数据表、哪个 API 和哪个前端入口。
7. `docs/superpowers/` 下较早的 specs/plans 可作为历史参考，但当前推进以 AGENTS、PRD/原型结构、当前路线图、验收计划、对象认领表和 `docs/design/` 活跃设计文档为准；如有冲突，以当前活跃文档为准。

## 2. 角色与权限入口原则

1. 前端顶层结构采用：`公开官网门户 -> 登录入口 -> 按权限进入对应业务后台`。
2. 不再使用“二期助手”作为后台一级入口或功能收纳口。
3. 不再把所有登录用户套进统一“角色工作台”模板；不同角色按实际功能数量和使用频率排布，只保持整体视觉风格一致。
4. 无权限后台入口直接不渲染，不显示“当前角色隐藏”、禁用菜单或占位列表。
5. 页面内无权限动作优先不展示；如因业务上下文必须展示，只能用简短只读状态说明。
6. 每个角色必须具备其日常任务所需的创建、查看、更新、审批、跟进、报表或治理入口，不允许只展示静态卡片。
7. 每个登录角色必须有自己的默认工作台总览页；总览页展示该角色的核心指标、待办、最近记录和功能入口，功能卡片与侧边栏导航必须指向同一套功能状态。
8. 测试账号只用于验收和演示，可以提供全权限和“演示视角”切换；它不代表生产环境中的普通账号或管理员可以随意切换角色，上线前应移除或关闭。

## 3. 角色功能边界

| 角色 | 默认入口 | 必须覆盖的核心任务 |
| --- | --- | --- |
| 游客/潜在客户 | 公开官网 + 客服 Agent | 了解公司、业务、政策、项目、活动、FAQ；发起咨询；完成活动报名 |
| 顾问 | 客户增长 | 手动录入客户线索、上传/粘贴客户资料、触发研判、查看推荐项目、创建跟进、创建任务、更新阶段、进入客户 360 |
| 员工 | 员工工作台 | 客户快捷录入/查询/状态更新、口述日报、组织架构查询、新人指南、受控只读查询 |
| 老师 | 学生服务工作台 | 请假审批、投诉反馈处理、心理辅助预警跟进、学业节点和申请进度查看 |
| 学生 | 学生服务台 | 提交请假、提交反馈、查询申请进度、查询学业考务、生活支持问答、心理倾诉 |
| 管理者 | 经营管理后台 | 增长总览、客户经营报告、员工日报汇总、学生心理健康周报、投诉处理周报 |
| 管理员 | 系统治理 | 用户、角色、权限、审计、通知、知识来源、系统状态、OpenAPI、seed、fallback；不承担顾问、员工、老师、学生、管理者的日常业务入口合集 |

## 4. 客户需求表覆盖原则

1. 客户需求表中的五类需求必须全部覆盖：客户研判、客服 Agent、企业智能助手、学生智能助手、智能报告。
2. 每个细分需求必须能追踪到：角色、前端入口、后端 API、数据库表、验收标准。
3. 顾问必须有手动录入客户线索功能，不得只依赖官网咨询、企业助手或 seed 数据。
4. 员工智能能力不得统一放入“二期助手”，而应拆入员工工作台、客户增长或学生服务工作台。
5. 学生智能能力不得统一放入“二期助手”，而应拆入学生服务台和老师学生服务工作台。
6. 客服 Agent 面向公开官网和潜在客户，负责公司信息、业务、政策、项目推荐、活动报名、FAQ 和轻量闲聊，不暴露内部 CRM 数据。
7. 智能报告主要归入管理者后台；管理员只在系统治理、审计或系统状态需要时查看相关配置和运行状态，老师只看与学生服务相关的报告。

## 5. 前端体验规则

1. 官网未登录首屏必须是公开官网门户，不允许直接进入后台。
2. 官网首页优先完成“建立信任 -> 解释服务 -> 引导咨询/项目/活动/登录”，不能做成后台功能陈列页。
3. 官网不得展示 CRM 客户列表、客户 360、员工日报、学生心理预警明细、审计日志、权限矩阵、OpenAPI、seed、接口健康等内部信息。
4. 官网 CTA 要区分“咨询/报名/查看项目”和“登录后台”。
5. 所有解释性文字尽量删除或精简，只保留必要的标题、字段、按钮、状态、风险提示、空状态和错误提示。
6. 业务页面不写项目阶段说明，不写“V2/V3 再做”等内部规划话术；登录页如需说明演示登录，只保留一句极短提示。
7. 后台页面要避免截图中出现的大面积空白、右侧长列表堆叠和禁用导航占位；功能少的角色使用紧凑布局，功能多的角色使用分区布局。
8. UI 效果图阶段必须至少输出 3 套整体风格方案；用户确认最终风格后再进入具体实现。
9. 后台页面必须支持刷新后保持当前业务位置；角色、后台模块、公开官网子页、关键业务对象 ID 应通过 URL、query/hash 或明确持久化策略恢复，不得只依赖 React 内存 state。
10. 前端交互不得停留在静态展示；搜索、筛选、漏斗节点、卡片、列表项、审批按钮、报名按钮等可点击元素必须触发明确业务结果：筛选列表、进入详情、提交接口、刷新数据或展示不可用原因。
11. 普通业务角色页面不得暴露实现话术，例如“真实 API”“fallback”“原型数据”“seed”“OpenAPI”“template_rule”“后续 V2/V3”等；管理员治理页可展示系统状态，顾问、员工、老师、学生、管理者页面只展示业务语言。
12. 每个功能区必须具备完整布局和真实数据承接：标题、指标、搜索/筛选、列表、详情、创建/编辑/处理入口、状态反馈、处理记录、空状态和错误状态按业务需要配置，不得只保留总览卡片。

## 6. 数据库原则

1. 当前可继续使用 SQLite，但最终表结构必须按客户、学生、日报、投诉、活动、研判、权限、报告等核心业务从零设计，并兼容后续 MySQL。
2. 新增能力优先新增表，不在无迁移策略时破坏既有表字段。
3. 关键业务表保留 `created_at`、`updated_at`；有状态流转的表必须保留状态字段和历史记录。
4. 所有写操作必须经过 service 层规则校验；AI 输出只能作为建议、结构化输入或草稿。
5. NL2SQL 只能做受控白名单只读查询，不允许直接执行任意写 SQL。
6. 学生心理预警只做辅助识别和跟进记录，不替代专业心理诊断。
7. 权限前端展示只用于体验；后端关键接口必须预留权限校验入口，并逐步落实接口级校验。

## 7. 架构边界

1. 后端采用 FastAPI + SQLAlchemy，继续按 `models/schemas/services/api` 分层。
2. API 层只做参数接收、依赖注入和服务调用；业务逻辑放在 `services`。
3. 数据访问优先通过 SQLAlchemy model 和 service 函数完成，不在路由中堆复杂查询。
4. 所有 API 统一返回 `{ code, msg, data }`，错误也使用同一结构。
5. 前端采用 React/Vite/TypeScript，请求统一经过 `frontend/src/api/client.ts`。
6. 知识库问答统一调用 Dify；未配置 Dify 时必须提供明确 fallback，不阻断 CRM、研判、活动、报告等主业务。
7. 客户画像、报告、心理预警、推荐等 AI 能力必须保留规则或模板兜底。

## 8. 执行规则

1. 代码或文档改动前先明确假设、取舍和成功标准；不清楚时先提问。
2. 简单优先，精准改动，只修改能直接追溯到当前任务目标的内容。
3. 大功能实现前先确认客户需求表、PRD、原型结构和阶段验收标准。
4. 每完成一个阶段必须运行对应验证命令。
5. 当前工作区是 Git 仓库，每次代码或项目文档改动完成后必须自动创建中文 commit。
6. 提交前先检查 `git status`，只 stage 当前任务相关文件。
7. 环境级 Git 警告 `C:\Users\97463/.config/git/ignore` 读取失败不影响本项目提交。
8. 多角色业务对象必须形成闭环：创建、列表、详情、状态更新、处理记录、发起方回显、处理方队列必须同时考虑；请假、反馈、成绩、日报、活动报名、CRM 跟进等对象不得只实现单角色提交或单按钮处理。
